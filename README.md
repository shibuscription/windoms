# Windoms README v0.1

プロジェクト名: **Windoms（Wind Orchestra Management System）**  
吹奏楽専用 管理OS

---

## 1. 目的とコンセプト

Windoms は、吹奏楽クラブの運営に必要な機能を統合する管理システム。
あわせて、保護者・役員の限られた時間/体力/気力を、できるだけ子どもたちの支援に向けられる状態を支える運営基盤とする。

- 日誌
- スケジュール
- シフト
- 会計
- 月謝管理
- 備品管理
- 楽譜・楽器管理

単機能ツールではなく、運営情報を横断して扱える基盤を目指す。

---

## 2. 基本思想

- 年度は 9/1 開始
- 現場負担を増やさない
- リアルタイム入力優先
- 紙運用は急に廃止しない（徐々に置換）

- 検索できることが価値

## 3. 用語定義（統一）

- **セッション（枠）**: スケジュールの基本単位
- 内部仕様・実装では `セッション` を使うが、一般利用者向け UI 表示では `セッション` を出さず、`予定` など自然な表現を使う
- **日誌**: 1日1枚の実績記録
- **活動ログ（タイムライン）**: 日誌内で時刻順に記録する実施内容

## 4. スケジュール管理（計画）

### 4.1 基本構造

- 管理単位は「セッション（枠）」
- 1日は 1〜複数セッションで構成
- 基本運用:
- 平日: 1セッション
- 土日: 基本2セッション
- 「午前/午後」は固定ラベルではなく、UI表示上で違和感なく扱う

TODO: 未決定（セッション名の最終表示ルール）

### 4.2 セッション管理項目

- 活動種別（通常 / 自主練 / イベント）
- 開始時刻 / 終了時刻
- 活動場所（第1/2/3音楽室 or 自由入力）
- 当番必要/不要

### 4.3 .ics 出力

- 月間スケジュール（セッション）から月単位で .ics を生成
- 1セッション = 1イベント

タイトル生成ルール:
- 通常: `【吹奏楽】{当番者名}`
- イベント日: `【吹奏楽】{当番者名} {イベント名}`
- 自主練: `【吹奏楽】自主練`

補足:
- 自主練の見守り担当は、初期仕様ではタイトルに含めない
- LOCATION はスケジュールの活動場所を使用
- イベント時は会場名などを LOCATION として設定可能

### 4.4 Today（日付ビュー）仕様

- Todayは「日付ビュー」として動作する
- URL仕様:
- `/today` は JST基準の本日を表示
- `/today?date=YYYY-MM-DD` は指定日を表示
- `date` が不正値（形式不正 / 実在しない日付）の場合は JST本日にフォールバックする
- 日付ナビUI:
- `← 前日` / `翌日 →` で日付移動し、URLクエリも更新する
- 日付表示（カレンダーアイコン含む）からミニカレンダーを開き、任意日へ移動できる
- ブラウザ戻る / 進むで日付遷移履歴を辿れる
- 日誌との連携:
- 指定日にセッションが1件以上ある場合のみ、当番日誌への導線を表示する
- 当番日誌導線の文言は常に `日誌へ` とする
- `日誌あり / 日誌なし` や `日誌を作成` の補助文言・別導線は採用しない
- 指定日に既存の日誌がある場合は、そのまま当番日誌画面へ遷移する
- 指定日に既存の日誌がない場合も、`日誌へ` 押下時に必要な日誌データを自動作成してから当番日誌画面へ遷移する
- 当番日誌画面左上の `Todayへ` は、現在開いている日誌の日付に対応する Today へ戻る
- `Todayへ` は常に今日の Today に戻る導線ではない
- 対象日が不明な場合のみ、安全な既存挙動へフォールバックしてよい
- Today / カレンダーから当番日誌へ入る導線は、日付文脈があるため引き続きその日の編集画面へ直接入ってよい

### 4.4.1 月カレンダー表示（DEMO）

- カレンダーは独立ルート `/calendar` で表示する。
- URLクエリは `ym=YYYY-MM`（表示月）と `date=YYYY-MM-DD`（選択日・任意）を扱う。
- カレンダー画面上部には固定タイトルを置かず、現在の年月 `YYYY/MM` を主見出しとして表示する。
- 前月 / 翌月ボタンは年月見出しの左右に配置する。
- 年月見出し `YYYY/MM` はクリック可能とし、押下で自前の年月選択UIを開く。
- 年月選択UIは標準 `input type="month"` を使わず、年切替と月選択から構成される軽量な自前UIとする。
- 「今日」ボタンは引き続き維持し、現在月へ戻る導線として使う。
- 768px を境に表示を切り替える。
  - 768px以上（PC）:
    - 通常練習/自主練は「開始時刻 + 種別名」を表示する。
    - イベントは「開始時刻 + イベント名」を表示する。
  - 767px以下（スマホ）:
    - 通常練習/自主練は「開始時刻のみ」を表示する。
    - イベントは「開始時刻 + イベント名（省略可）」を表示する。
- セッション表示は2行構成とする。
  - 1行目: 時刻と種別名/イベント名
  - 2行目: 当番/見守り（苗字）。未定は `-` を表示する。
- DEMOの当番/見守り表示名は、以下の苗字ホワイトリストのみを使う。
  - `瀬古 / 大滝 / 中村 / 今井 / 青木 / 水野 / 渋谷 / 熊澤 / 加藤`
- DEMOデータにホワイトリスト外の当番/見守り名が残っている場合は、`-` にはせず、ホワイトリスト内の自然な苗字へ置換する。
- DEMOでは、出欠・TODO担当・詳細モーダル等を含む人物名表示も、ホワイトリスト内の苗字に統一する（関係付き表記は使わない）。
- 外部講師名は例外とし、DEMOでも `井野` の表示を許容する（ホワイトリスト置換対象外）。
- 日付セルタップは、その日の日単位ボトムシート（DaySheet）を開く。
- セル内セッション行は表示専用とし、個別タップ判定は持たない。
- DaySheetはPC/スマホとも共通コンポーネントを使う。
  - 767px以下（スマホ）: 下からスライドイン
  - 768px以上（PC）: 幅を絞った中央表示（フェード＋軽い上昇）
- DaySheet内は当日セッションを縦カードで表示し、各カード左のカラーラインで種別を表現する。
- DaySheet内アクションとして `Todayへ` / `日誌へ` / `＋` を表示する（DEMOでは準備中操作を許容）。
- DaySheet内のカードデザインはToday画面のセッションカード（ミニ版）に寄せる。
  - 種別ごとの薄い背景色
  - 右上の種別バッジ
  - 時刻 / 当番・見守り / 場所 / 出欠ミニ表示
- 種別は右上バッジのみで表現し、本文に種別テキスト行は出さない。
- 編集/削除アイコンは右上バッジ付近に配置し、カード下部に専用行を作らない。
- DaySheetでは注意書き枠を表示しない。
- カレンダーセル2行目のラベル表示は端末幅で切り替える。
  - 767px以下（スマホ）: 苗字のみ（`当番:` / `見守り:` ラベルは非表示）
  - 768px以上（PC）: 通常練習/イベントは `当番: 苗字`、自主練は `見守り: 苗字`
- DaySheet内の出欠ミニ表示はタップで出欠モーダルを開く。
  - DEMOでは表示のみとし、保存は行わない。
- ボタン意味は固定する。
  - `今日`: カレンダー表示を今日へ戻す（画面遷移しない）
  - `Todayへ`: Today画面へ遷移する

### 4.5 Today備考（notice）表示仕様

- `scheduleDays.notice` はToday上部に表示する（改行反映）
- `scheduleDays.notice` は計画側の連絡事項（Today備考）として扱う
- 初期は3行相当で折りたたむ
- 3行を超える場合のみ、`▼ ひらく / ▲ たたむ` トグルを表示する
- 展開はインラインで行い、全文を読めるようにする
- 展開時は備考エリアが見失われにくいようにスクロール補正する
- 展開後も最大高さを設け、超過分は備考エリア内スクロールで読む

### 4.6 共通メニュー（オーバーレイ）

- 画面右上にメニューアイコン（`☰`）を配置する
- メニュー押下でフルスクリーンのオーバーレイメニューを表示する
- オーバーレイ上部にJSTの今日の日付を大きく表示する（曜日表示を含む）
- 日付文字は暗背景上で白系を基本とし、曜日のみ色分けする（平日=明るいグレー、土=青系、日=赤系）
- 上部の日付ヘッダを押すと Today 画面へ遷移する
- ヘッダ下にスマホホーム画面風のアイコングリッドを表示する
- メニューは以下の3セクション構造とする（見出しは小さめ淡色）:
- 運用: Today / 日誌 / カレンダー / TODO
- 活動予定: 活動予定 / 出欠 / 見守り
- 管理: 会計 / 楽器 / 楽譜 / 資料 / メンバー / リンク集 / 設定
- MENUはカテゴリごとに独立したセクション（見出し + 個別grid）で表示する
- 確定カテゴリは「活動 / 会計 / 資産 / 設定」の4つとする
- そのカテゴリ内に表示対象タイルが1つも無い場合、カテゴリ自体を表示しない
- 当番可否アンケートは月1回イベントのため、専用メニューは追加しない
- 気づき導線として、運営向けに「活動予定」メニューへ未回答バッジ（SURVEY_OPEN中のみ）を表示する
- 気づき導線として、運営向けにホーム上部へ「今月のTODO」カードを状態優先度で1枚表示する
- 一般メンバーは当月の活動予定確認を基本「カレンダー」から行う
- 各セクション内のグリッドはレスポンシブで自動列数調整（スマホ3列前後、PCはより多列）とする
- メインメニューのタイルは正方形固定とし、横幅はレスポンシブで列数のみ増減させる（PCでは4列以上を許容、スマホは3列を基本）
- メニューは全幅ランチャーとして表示し、縦スクロールは `menu-overlay` の1箇所に統一する（スクロールバーは表示領域の右端）
- メニュータイルのグリッドは左寄せ配置とする
- DEMOではDEV PANELで表示ロール（`child` / `parent` / `admin`）を切り替えられるようにする
- メニューはロールに応じて「見せない」方式で制御し、権限外モジュールは表示しない
- 日誌/TODOは将来思想として、大人向け（当番日誌 / TODO）と子ども向け（練習日誌 / 宿題）を別モジュールとして扱う
- メニュータイルは正方形（1:1）に統一し、列数は固定しない（`auto-fill + minmax`）
- オーバーレイ内に白い大枠コンテナは置かず、暗背景上に直接グリッドを配置する
- アプリ名ラベルは原則1行表示（改行させない）
- 旧「スケジュール」は削除し、「カレンダー」「活動予定」に分割する
- Dock（下部固定タブ）は採用しない
- 閉じ方は `×` ボタンを基本とし、背景クリックで閉じる（Esc閉じるはPCで許容）
- DEMO段階では、未実装モジュールは仮遷移（viewパラメータ等）で導線のみ確保する

## 5. 日誌モジュール（実績ログ）

### 5.1 基本構造

- 日誌は **1日1枚**
- 当番日誌はセッション単位ではなく日単位で扱う
- 1日に複数セッションがある場合でも、当番日誌はその日1枚の扱いとする
- 日誌の `notes` は実績側の記録メモ（日誌メモ）として扱う
- 1日共通情報:
- 活動時間（通し）
- 活動場所（1つ）
- 講師（メイン指導者）はセッション単位で扱う
- 外部講師（1日単位）
- 先輩（1日単位）
- セッション別に分ける主対象:
- 当番
- 出欠（人数 + 欠席者名任意）
- 日誌画面には Today備考（`scheduleDays.notice`）を表示しない
- 当番日誌画面の活動場所欄では、その日に存在する各セッションを必ず表示する
- 活動場所が設定されているセッションは場所名を表示し、未設定セッションも時間付きで行を出したうえで `-` など未設定と分かる表示にする
- `minamigaoka` の当番日誌は Firestore を正として日単位で正式保存する
- メニューから当番日誌を開く場合は、直接編集画面ではなく月別の当番日誌一覧画面を開く
- 当番日誌一覧画面は月単位で表示し、初期表示は当月とする
- 当番日誌一覧画面の表示対象は、その月のうちセッションが1件以上ある日だけとする
- 一覧には日誌作成済み日だけでなく、セッションありかつ未作成の日も表示する
- 一覧は日付降順のレスポンシブなタイルレイアウトとする
- PC では 3 列程度を基準とし、画面幅に応じて 2 列 / 1 列へ自然に切り替わるようにする
- 各カードには日付、作成済み / 未作成、セッション情報、当番表示を出す
- 作成済み / 未作成はバッジと背景色の両方で分かるようにし、作成済みは白、未作成は薄いグレーを基本とする
- カード右上には作成済み / 未作成バッジのみを表示する
- セッション種別やイベント情報はカード本文側でバッジ表示する
- セッション種別の表記は `通常` ではなく `通常練習` とする
- 一覧カードの当番表示は、日誌画面下部の当番ハンコ表示を縮小した閲覧専用表示とし、捺印済み / 未捺印 / 当番未設定の見た目ルールも日誌画面に揃える
- 一覧カードの当番ハンコ表示名も、当番の表示単位に合わせて Family 単位で扱う
- 月移動ボタンは通常状態で文字が読める視認性を確保する
- 一覧カード押下時は、その日の日誌が未作成なら自動作成してから編集画面へ遷移する
- 当番日誌編集画面からは、その日付が属する月の当番日誌一覧画面へ戻れるようにする
- 当番日誌画面は原則自動保存とし、明示的な保存ボタンは置かない
- 当番日誌画面の右上には保存状態表示を置き、少なくとも `保存中` / `保存済み` / `保存失敗` を表示できるようにする
- 保存状態表示は見た目だけではなく、実際の Firestore 保存結果に基づいて更新する
- 当番日誌画面の見出しは `講師` とする
- 当番日誌画面の `講師` 欄は、各 session に保存されたメイン講師予定を本文内で参照表示する
- 表示は各セッションごとに `○ / × / -` とし、未設定は `-` で分かるようにする
- メイン講師予定のための別モーダルや `予定を見る` 導線は置かない
- 天候 / 出欠 / 講師在席チェック / 外部講師・先輩の追加削除 / 活動記録の追加編集削除 / 当番ハンコは、操作確定時に Firestore 保存してよい
- 備考欄だけは例外として、入力中には保存せずフォーカスアウト時に保存する
- 備考欄は変更がある場合のみフォーカスアウト時に Firestore 保存する
- 他画面の保存方式は今回変更しない
- Today備考（計画側）と日誌メモ（実績側）は別フィールド・別概念として扱う
- Today備考と日誌メモは相互に同期しない
- 当番ハンコは「予定表示」ではなく「捺印（実績）」で確定する
- 予定当番（計画）と捺印実績（実績）は別概念として扱い、強制同期しない
- 講師（メイン指導者）の予定（計画）と実績（日誌）は別概念として扱い、強制同期しない

### 5.2 活動ログ（タイムライン）

- 1日通しで入力
- 時刻順で並ぶ（自動ソート）
- 開始時刻のみ（終了時刻不要）
- 5分刻み
- 初期値は現在時刻に最も近い5分
- 入力はリアルタイム前提（後日入力最適化は不要）

### 5.3 活動種別と曲名

活動種別:
- 筋トレ
- 腹式呼吸
- 基礎練習
- 個人練習
- 合奏
- 休憩
- その他（自由入力）

曲名:
- 合奏は曲名入力あり
- 個人練習も曲名を任意で入力可能
- 曲は複数選択を許容（`songIds`）
- 表示は開始時刻ベースで、必要に応じて「種別（詳細 / 曲名）」を併記する

### 5.4 検索価値

- 曲名検索（初回練習日など）
- 先輩検索（来訪履歴）
- パート検索
- 種別検索

日誌は「練習履歴データベース」として機能する。

### 5.5 実績入力UI方針（講師・先輩）

- `actualInstructors` / `actualSeniors` は配列UIで編集する（カンマ区切り入力は採用しない）
- 追加済みは行リストで表示し、各行の削除で1要素ずつ配列から除去する
- 追加はインライン入力（`＋追加`）で行う
- 外部講師・先輩の入力は自由入力を基本とする
- 候補は入力欄一体型のドロップダウンで表示し、履歴から再利用できるようにする
- 候補優先順位は `planned（未追加分）` → `履歴（過去 dayLogs の actual）` → `自由入力` を基本としてよい
- 先輩欄では、必要に応じて member 上の `obog` 候補を補助表示してよいが、member 選択必須にはしない
- 候補クリックで即追加する
- 空文字（trim後）は追加しない
- 重複値は許容する（配列としてそのまま保持）

### 5.5.1 講師（メイン指導者）の予定/実績

- メイン講師は単数で扱う（現状: 井野先生）
- 在席はセッション単位で扱う
- 予定はスケジュール側の `sessions.mainInstructorPlanned` を参照する
- `mainInstructorPlanned` は、カレンダーのセッション新規作成 / 編集モーダルで設定する
- 値はメイン講師がそのセッションに来る予定かどうかを表し、`true / false / null(or未設定)` を許容する
- 今回は最小実装として、カレンダー本体や Today には表示せず、当番日誌画面用の参照値として扱う
- 将来、シフト作成画面側でより詳細な講師予定管理へ拡張できる余地を残す
- 実績は日誌側でセッション単位チェックとして記録する
- 実績チェックの初期状態は全セッション未チェック
- 予定がOFFでも実績をONにできる
- 予定がONでも実績をOFFにできる
- 予定と実績は同期しない
- メイン講師は外部講師として扱わない

### 5.6 活動記録UI（一覧・モーダル）

- 一覧は「開始時刻～ / 種別（補足・曲名）」で表示する
- 開始時刻はゼロ埋めしない（例: `9:05`）
- 一覧表示は時刻列を固定幅・右寄せにし、`～` の位置を縦に揃える
- 4桁時刻（例: `16:35～`）でも時刻表示は改行しない
- 活動記録モーダル保存時、`種別` は必須とする
- `種別` 未入力で保存した場合は、保存せずエラー表示する
- エラー表示は `種別` ラベル行の右側に表示する（ドロップダウンに隠れない位置）

### 5.7 活動記録の曲追加UI（DEMO）

- `songIds` は `string[]` で保持し、最大10件
- 曲追加はサジェスト候補タップまたは Enter で行う
- 当番日誌の活動記録追加モーダルにおける曲名候補は、Firestore ルート直下の `scores` コレクションを参照する
- 曲名候補の取得元は、楽譜モジュールと同じ正データ `scores` に統一する
- ダミー候補や固定配列を正データとして使わない
- 曲名欄はインクリメンタルサーチで候補を絞り込めるようにする
- 候補絞り込みは、可能な範囲で楽譜一覧の検索正規化と揃える
- 曲追加UIのアクションは `キャンセル` のみ（`追加` ボタンは置かない）
- 入力文字列は `songMaster.name` と一致する場合のみ追加する（trimして判定）
- 同一曲の重複追加は行わない
- 曲が解決できない場合や重複追加時は、入力欄上にローカルトーストでフィードバックを表示する
- トーストは短時間（約2秒）で自動消去し、連続発生時は上書きする
- 曲候補の読込中 / 候補なし / 読込失敗時も、入力欄付近で分かるようにし、無反応に見える状態を作らない

### 5.8 当番ハンコ（捺印実績）

- 初期表示は未捺印（薄グレー）
- 未捺印時は「予定当番名（計画）」を表示する
- 予定当番が未設定の場合は空欄表示でよいが、捺印操作自体は可能とする
- タップで捺印し、赤色表示へ切り替える
- 捺印済みは「捺印者名（実績）」を表示する
- 捺印時に軽いスタンプ演出を行う（過度な演出はしない）
- 他人枠をタップした場合は Confirm を表示し、承認時のみ上書き捺印する
- 同一捺印者が再タップした場合は無反応にせず「捺印済み」等でフィードバックする
- 当番ハンコは、押したユーザー本人名ではなく、そのユーザーが所属する Family 名で保存・表示する
- 当番の表示単位と当番ハンコの表示単位は Family 単位で揃える
- 既存の本人名ベースの捺印データが残る可能性がある場合は、表示側で後方互換を考慮する
- 今回の Firestore 永続化対象は当番日誌に限定し、他画面・他モジュールの保存方式は別途検討とする

## 6. 出欠管理

### 6.1 役割

- 出欠モジュールは、月単位でセッションごとの出欠を登録・一覧確認するためのモジュールとする。
- 対象は部員の出欠とする。
- 回答単位はイベント単位ではなくセッション単位とする。
- 既存の Today / カレンダー / DaySheet / 当番日誌などに表示する出欠の入力元として扱う。
- 締切管理や通知送信そのものは主責務としない。
- 月間スケジュールが固まった後に出欠登録を始め、その後も随時更新可能な前提で運用する。
- 通知や回答依頼の強化は将来拡張としてあり得るが、初期段階では必須としない。
- シフト作成などの下流処理のために厳密な締切を課す仕組みは、現時点では不要とする。

### 6.2 出欠対象と回答者

- 出欠対象は部員のみとする。
- 回答できる人は以下とする。
  - 部員本人
  - その部員に紐づく保護者
  - 管理者
- 部員本人は自分自身の出欠を回答できる。
- 保護者は、自分に紐づく部員全員分を回答できる。
- 管理者は全員分を追加・編集できる。

### 6.3 回答単位と回答値

- 回答単位はセッション単位とする。
- 回答値は `○ / △ / × / コメント` を持つ。
- 記号の意味は以下とする。
  - `○` = 参加
  - `△` = 条件付き参加 / 未確定
  - `×` = 不参加
  - `コメント` = 自由記述
- 未回答状態は存在してよい。
- 新規月や途中追加されたセッションでは、未回答があり得る。
- ただし「未回答のまま明示的に保存する操作」は必須ではない。
- 回答する場合は `○ / △ / ×` のいずれかを選ぶ。

### 6.4 閲覧権限

- 出欠一覧は、全員が全員分を閲覧できる前提とする。
- 自分の分だけでなく、全体を一覧できることを重視する。
- 管理者向けには、未回答者を把握しやすい補助表示を持ってよい。

### 6.5 月切り替え

- 出欠モジュールは月単位で表示する。
- UI はカレンダーモジュールに近い操作感を目指す。
- 画面表示上の用語は `予定` に統一し、`セッション` という文言は UI に出さない。
- 年月表示は `YYYY/MM` とする。
- 前月 / 今月 / 翌月に移動できる。
- 表示対象は、その月のセッション一覧を基本とする。

### 6.6 一覧画面の基本表示

- 一覧画面は、まず `行 = セッション / 列 = 部員` の表形式を標準表示とする。
- 目的は、今月分の全員の出欠を一目で一覧できることとする。
- 画面表示は `予定` 系の運用思想を保ちつつ、通常練習 / 自主練の行ヘッダや列ヘッダに不要な `予定` 文言は出しすぎない。
- イベントのみ、必要に応じてイベント名表示を維持してよい。
- 各セッション行の見出し付近には集計表示を出す。
- 例: `○0 △0 ×0 ー9`
- ここで
  - `○` = 参加人数
  - `△` = 条件付き参加 / 未確定人数
  - `×` = 不参加人数
  - `ー` = 未回答人数
- コメントは一覧表には表示しない。
- 編集権限がある場合は、一覧グリッドの各セルからその1件だけを直接変更できてよい。
- 直接編集は軽いモーダルで行い、状態選択と確定を最小構成とする。
- 直接編集でも、既存の出欠保存構造と整合する既存保存処理を再利用する。
- 編集UIの選択肢は `○ / △ / ×` を基本とし、利用者が明示的に `未回答` を選ぶUIは置かない。
- 既存データとして未回答が存在すること自体は許容する。
- 入力運用上、未確定は `△` を用いる。

### 6.7 行列入れ替え表示

- 一覧画面は、標準表示に加えて `行 = 部員 / 列 = セッション` の入れ替え表示を持つ。
- 目的は、特定の部員ごとの出欠状況を見やすくすることとする。
- 初期表示は標準表示（`行 = セッション / 列 = 部員`）とする。
- 表示切り替えUIは、`行と列を入れ替え` という単一ボタンで扱ってよい。
- 表示モードは localStorage に保存し、前回選んだ状態を次回も引き継ぐ。

### 6.8 ヘッダ固定

- 一覧表は、横スクロール・縦スクロールの両方が発生し得る前提でヘッダ固定を行う。
- 標準表示では
  - 上部の部員ヘッダを固定
  - 左端のセッション情報列を固定
- 行列入れ替え表示でも
  - 上部ヘッダを固定
  - 左端ヘッダを固定
- スクロールしても、今どの部員・どのセッションを見ているか分からなくならないことを優先する。

### 6.9 初期スクロール位置

- 一覧にスクロールが必要な場合は、本日分または直近の予定が頭出しされることを優先する。
- 今月の中で、今日または最も近い将来のセッション付近が最初に見えることを優先する。
- 厳密なアルゴリズムは実装時に自然な範囲で決めてよい。

### 6.10 個別閲覧 / 編集モーダル

- 一覧画面では、部員ヘッダ名または部員行名をクリックすると、その部員の個別モーダルを開く。
- その部員について編集権限がある場合は編集モーダルを開く。
- 編集権限がない場合は閲覧モーダルを開く。
- モーダルのタイトルは説明文を付け足さず、部員名のみでよい。
- モーダル内はカードを縦に並べるより一覧性を優先し、`1予定 = 1行` の表形式寄りUIを基本とする。
- 各行には `日付 / 時間 / 種別バッジ / イベント名 / 状態 / コメント` を自然に確認できる構成を持たせる。
- 各行ヘッダの見せ方は、メイン一覧の予定ヘッダと可能な限り揃える。
- 通常練習 / 自主練では不要な `予定` 文言を出さず、イベント時のみ必要な名前表示を行ってよい。
- コメントは常設入力欄ではなく、アイコン起点の補助UIで扱ってよい。
- 閲覧モードでは、コメントがある場合のみコメント閲覧アイコンを表示してよい。
- 編集モードでは、コメントありは編集アイコン、コメントなしは追加アイコンで扱ってよい。
- コメントアイコン押下時は、小モーダル等で閲覧 / 編集できればよい。
- セッション数が多い場合に備え、モーダル内は縦スクロール可能とする。
- 閲覧モーダルでは、コメントも含めてその部員1人分の出欠を確認できる。
- コメントは一覧表には表示しないが、個別閲覧 / 編集モーダルでは閲覧できる。
- コメントの閲覧範囲は全員可とする。
- 編集モーダルでは、回答可能な部員について同じ構造で編集できる。
- 編集可能範囲は以下とする。
  - 管理者: 全員分
  - 部員本人: 自分自身
  - 保護者: 紐づく部員分
- スマホでは、一覧と個別モーダルの予定ヘッダを月日 / 時刻中心の省スペース表示に寄せてよい。

### 6.11 既存の出欠表示との関係

- このモジュールは、既存の Today / カレンダー / DaySheet / 当番日誌などに表示する出欠の入力元とする。
- 既存画面の出欠表示と別管理の新しい概念を増やすのではなく、出欠モジュールで登録した内容を元データとして扱う。
- 既存の出欠表示は、将来的にこのモジュールと接続される前提で整理する。

### 8.4 会計モジュール（Firestore正式化フェーズ）

#### ■ 基本思想

- 会計は「団体の財布・口座が実際に動いたとき」にのみ記録する（現金主義）。
- 立替（建替）は会計モジュール本体とは分離し、別モジュールとして扱う。
- 会計は団体の財布・口座が実際に動いた事実のみを記録する。
- 購入依頼・立替・お弁当は会計本体とは別の業務モジュールとして扱う。
- 周辺モジュールから会計への連携は片方向とする。
- 会計へ支出レコードを追加した後は、周辺モジュール側との逆同期は行わない。
- 周辺モジュール側の状態変更により、既に作成済みの会計レコードを自動で更新・削除しない。
- 今回の会計トップは DEMO ダミーデータではなく Firestore を正として表示する。

---

#### ■ 取引の種類

以下の3種類のみを扱う。

- 収入
- 支出
- 振替（口座間移動）
- Firestore 上の `accountingTransactions.type` は `income` / `expense` / `transfer` の3種のみとする。

---

#### ■ 口座

- 口座は Firestore 管理とする。
- 将来的に口座追加・編集が可能な前提で構造を作る。
- 口座マスタは `accountingAccounts` で管理する。
- 口座は少なくとも `name`, `sortOrder`, `isActive` を持つ。
- 口座の表示順は `sortOrder` 昇順、同値または未設定時は `name` 昇順、最後に `id` で安定化する。
- 口座一覧、会計トップの口座一覧、取引登録の口座選択、会計期開始時の期首残高入力順は、原則この並び順に従う。
- 口座の期ごとの情報は `accountingPeriodAccounts` で管理する。
- `accountingPeriodAccounts` は少なくとも `periodId`, `accountId`, `openingBalance` を持つ。
- 初回セットアップ時、口座が1件も存在しない場合は以下の3口座を初期候補として扱う。
  - 現金（会計手元金）
  - 現金（会長手元金）
  - ゆうちょ銀行
- これらの初期候補は隠し固定値として自動生成するのではなく、利用者が確認・編集できる口座マスタとして扱う。
- 各口座は期ごとに「期首残高」を持つ。
- 残高は保存値ではなく取引集計から自動計算する。

---

#### ■ 科目（DEMO仕様）

- 科目は固定とする（UIからの追加・編集は行わない）。
- 将来のバージョンアップで科目マスタ機能を実装できるよう、取引データは科目名ではなく「科目ID」を保持する設計を前提とする。
- 固定科目は差し替え可能な構造にする。

---

#### ■ 期（年度）

- Windoms の会計年度は 9 月 1 日開始とする。
- 会計は期（年度）単位で管理する。
- トップ画面では「現在の期」のみを扱う。
- 期の状態は `editing` / `closed` の2状態で管理する（ロック型）。
- 簡単な期切替機能は設けない。期管理ページで現在期と過去期を閲覧する。
- `accountingPeriods` で会計期を管理する。
- `state` が `editing` の期を現在の期とみなす。
- `editing` は常に1件だけとする。
- 会計トップは現在の期のみ表示する。
- `closed` は過去期として扱う。
- 最初の会計期を開始するときは、年度と各有効口座の期首残高を入力して作成する。

---

#### ■ 年度締め・繰越

##### 年度を締める

- `editing` の現在期のみ、年度締めを実行できる。
- 締め処理時に現在期を `closed` に変更し、以後は仕訳不可（閲覧のみ）とする。
- 締め処理時に新しい `editing` 期を1つ生成する。
- 各口座の期末残高を、新期の期首残高へ自動設定する（繰越）。
- 期首残高は繰越で確定し、以後は直接編集しない（修正は仕訳で行う）。

- 残高の正（Single Source of Truth）は「期首残高 + 取引明細の合算」とする。
- 現在残高はこの合算結果から算出する。
- 将来的に表示高速化のためのキャッシュを持つ可能性はあるが、常に再計算可能な設計とする。
- 残高計算式は `期首残高 + 収入 - 支出 + 振替入金 - 振替出金` とする。
- 年間取引件数は多くない想定のため、当面は画面表示時集計でよい。

---

#### ■ 会計トップ画面（現在の期）

思想：ダッシュボード専用（入力と状況把握を分離）。

- Windoms の会計年度は 9 月 1 日開始、8 月 31 日終了とする。
- `2025年度` は `2025-09-01` から `2026-08-31` を指す。
- 会計画面の日付表示と日付入力初期値は、常に `YYYY-MM-DD` の日付のみを使い、ISO文字列や Timestamp の生値をそのまま見せない。
- 会計トップはダッシュボードとして扱う。
- 現在の期のみ表示する。
- 上段：主要操作ボタン（収入 / 支出 / 振替）を常時横3列で表示する。
- 中段：口座一覧（縦リスト）を表示し、左に口座名・右に残高を配置する。
- 下段：今期の収支（収入合計 / 支出合計 / 差額）を表示する。
- 補助領域：月別支出グラフを表示する。
- 会計関連で期内の月を並べる表示は、原則として会計年度の開始月から順に表示する。
- 月別支出グラフは `9月, 10月, 11月, 12月, 1月, 2月, 3月, 4月, 5月, 6月, 7月, 8月` の順で表示する。
- 期管理フォーム（年度締め・期首残高編集）は会計トップに置かない。
- `editing` の期が存在しない場合は、会計データ未設定と分かる自然な空状態を表示する。
- Firestore 読み込み中は無反応に見えないローディング表示を出す。
- エラーは会計トップ画面内の分かりやすい位置に表示する。
- 収支計算書は閲覧用途のため、管理者以外にも導線を表示してよい。

---

#### ■ Firestore データ構造

- 会計モジュールでは以下のコレクションを使用する。
  - `accountingPeriods`
  - `accountingAccounts`
  - `accountingPeriodAccounts`
  - `accountingTransactions`
- `accountingTransactions` で取引を管理する。
- 収入・支出は `accountId` を持つ。
- 振替は `fromAccountId` と `toAccountId` を持つ。
- 収入・支出は `categoryId` を持つ。
- 振替には `categoryId` を持たせない。
- `accountingTransactions` は少なくとも `date`, `amount`, `memo`, `periodId`, `createdAt`, `updatedAt` を持つ。
- `amount` は常に正の値で保持し、増減は `type` に応じて解釈する。
- `memo` は既存の自由記述欄ルールに合わせ、プレーンテキスト保存とし、表示時のみ改行保持と `http/https` URL 自動リンク化を行う。

---

#### ■ 会計科目

- 会計科目は当面固定定義とする。
- 科目体系は、既存運用で使用している一覧を維持する。
- 取引保存は引き続き `categoryId` ベースで行う。
- 科目追加編集 UI は現時点では設けない。
- 将来的に必要になった場合にマスタ化を検討する。
- 会計科目はカテゴリと項目の 2 階層構造を持つ。
- 入力画面ではカテゴリごとにグループ化して表示する。
- カテゴリ見出し自体は選択不可とし、個別科目のみ選択可能とする。
- 科目の表示順は、既存運用で使用しているカテゴリ順・項目順を維持する。
- 収支計算書のカテゴリと項目は、この会計科目定義をそのまま使う。
- 入力画面の科目体系と収支計算書の集計・表示構造は一致させる。
- 科目体系を変更する場合は、入力画面と収支計算書の両方への影響を前提に扱う。

#### ■ 旧会計 CSV からの初期データ移行

- 旧会計 CSV からの初期データ移行時は、旧 CSV をそのまま 1 行 1 取引で投入せず、口座間移動を `transfer` へ正規化してから Windoms 用データへ変換する。
- 旧 CSV 上の `前年度より繰越` は、各口座の期首残高として扱う。
- 旧 CSV 上で同日・同額・片側が収入、片側が支出で、摘要や科目から `振替` / `引出` / `預け入れ` 等の対応関係が取れる口座間移動は、1 件の `transfer` に統合する。
- 無理に自動統合して誤るより、未統合候補として残すことを優先する。
- 科目体系は既存の固定定義を優先し、旧 CSV の科目名を可能な限り既存 `categoryId` に対応づける。
- 変換結果は一度 JSON などの中間成果物で確認してから投入する運用とし、変換スクリプトから本番 Firestore へ直接書き込まない。
- `2025年度` の移行対象期間は `2025-09-01` から `2026-08-31` とする。
- 旧 CSV での口座対応は以下を基本とする。
  - `現金出納帳` → `現金（会計手元金）`
  - `預金出納帳` 内の `ゆうちょ銀行` → `ゆうちょ銀行`
  - `預金出納帳` 内の `会長手元金` → `現金（会長手元金）`
- 変換スクリプトは少なくとも以下を出力できるようにする。
  - `period`
  - `accounts`
  - `openingBalances`
  - `normalizedTransactions`
  - `unmatchedTransferCandidates`
  - `categoryMappingResults`
  - `summary`
- 変換プレビューは `minamigaoka` で `powershell -ExecutionPolicy Bypass -File .\scripts\normalize-accounting-csv.ps1 -CashCsv <現金CSV> -DepositCsv <預金CSV> -FiscalYear 2025 -OutJson <出力JSON> -OutMarkdown <出力MD>` のように実行できるようにする。
- 会計初期データの Firestore 投入は、正規化済み preview JSON を確認したうえで行う。
- 投入前に、対象年度・対象口座・期首残高件数・収入件数・支出件数・振替件数を確認する。
- 同一年度の `accountingPeriod` がすでに存在する場合は、原則として重複投入しない。
- 既存データがある状態で再投入する場合は、自動上書きを行わず、手動確認を前提とする。
- 投入スクリプトは dry-run を既定とし、明示的な実行指定がある場合のみ Firestore へ書き込む。
- `2025-08-28` の `印鑑ケース` 支出 1 件は移行時例外として扱い、投入時に日付を `2025-09-01` へ補正して `2025年度` データへ含める。
- Firestore 投入は `minamigaoka` で `node --env-file=.env ./scripts/import-accounting-migration.mjs --input <preview-json>` を dry-run として実行し、問題なければ `--apply` を付けて投入する。
- 会計の危険操作（期削除・口座削除）は通常 UI には置かず、初期移行補助が必要な場合だけ管理用スクリプトで扱う。
- 初期移行前の空データ掃除は、対象年度の `accountingPeriods` / `accountingPeriodAccounts` / `accountingTransactions` を dry-run で確認したうえで、管理用削除スクリプトで行う。
- 口座マスタ `accountingAccounts` は今回の初期移行では削除せず、既存の `accountId` / `name` / `sortOrder` / `isActive` が一致するものを importer が再利用する。
- 空データ掃除と再投入の基本手順は以下とする。
  1. 削除スクリプト dry-run
  2. 削除スクリプト本実行
  3. importer dry-run
  4. importer 本実行
- 削除スクリプトは日常運用用 UI の代替ではなく、初期移行補助用途の管理手段としてのみ使う。

#### ■ 収入・支出 / 振替モーダル

- 収入モーダルの口座項目ラベルは `入金口座`、支出モーダルの口座項目ラベルは `出金口座` とする。
- 収入・支出モーダルの項目順は、`入金口座 / 出金口座` → `レシート画像（任意・複数）` → `日付` → `金額` → `科目` → `摘要` とする。
- `添付画像` というラベルは使わず、`レシート画像（任意・複数）` に統一する。
- 収入・支出では `メモ` ではなく `摘要` を使い、摘要は必須とする。
- 摘要入力は、同種別の過去取引の入力履歴を候補表示するインクリメンタルサーチ付き入力とし、候補がなくても手入力できるようにする。
- 収入は過去の収入摘要、支出は過去の支出摘要を候補元とする。
- 振替モーダルの項目順は、`日付` → `出金口座` → `入金口座` → `金額` → `備考` とする。
- 振替では `メモ` ではなく `備考` を使い、備考は任意、候補提示は行わない。
- 振替の出金口座と入金口座が同一の場合はエラーとする。

#### ■ 口座通帳画面

- タイトルは `口座通帳：{口座名}` とし、対象口座名を見失いにくくする。
- 年度表示は別行または補助情報として表示してよい。
- 口座通帳画面にも会計トップと同様の `収入 / 支出 / 振替` ボタンを置いてよい。
- 口座通帳から登録モーダルを開いた場合は、現在表示中の口座を初期値としてセットする。
- 収入は入金口座、支出は出金口座に現在口座を初期値セットする。
- 振替は出金口座に現在口座を初期値セットし、入金口座は未選択で開く。
- これらは初期値であり固定値ではなく、ユーザーは必要に応じて変更できる。
- 口座通帳の明細表は、少なくとも `日付 / 種別 / 科目 / 摘要・備考 / 収入金額 / 支出金額 / 残高 / 操作` の列を持つ。
- 収入と支出は別列で表示し、支出はマイナス記号付きで表示しない。
- 振替は、対象口座に対して入金なら `収入金額` 列、出金なら `支出金額` 列に表示する。
- 口座通帳では、会計登録可能な権限を持つユーザーに対して、明細の編集・削除を可能としてよい。
- 編集は既存の登録モーダルを編集モードで再利用し、収入は収入モーダル、支出は支出モーダル、振替は振替モーダルを使う。
- 削除は確認ダイアログを出してから実行する。
- 会計取引に作成ユーザ情報が無い場合は、管理者以外でも通常の削除確認のみでよい。
- 口座通帳では日付列ヘッダで昇順 / 降順の切替をできるようにする。
- 口座通帳一覧の既定表示は、日付の古い順とする。
- 同一日付内では、早く作成した明細が先に見える順とし、`createdAt` 昇順を基本に安定した補助キーで並びを保つ。
- 逆順表示時は、`繰越` 行の位置を調整するなど、見た目が不自然になりすぎない表示を優先する。
- 口座通帳は、PC とスマホで表示形式を分ける。
- PC ではテーブル形式を基本とし、一覧性を優先する。広めのコンテナ幅を使ってよく、`日付 / 種別 / 科目 / 収入金額 / 支出金額 / 残高 / 操作` は原則として折り返しを避ける。
- PC では `摘要 / 備考` のみ必要に応じて折り返してよい。
- スマホでは表の縮小表示ではなくカード形式を基本とし、1件ずつ追いやすいことを優先する。
- スマホのカードでは、少なくとも `種別 / 日付 / 金額 / 摘要・備考 / 科目 / 残高 / 編集・削除` を自然な優先順位で表示する。
- 期首残高もスマホではカードとして自然に表示してよい。

## 14. 設計原則（詳細）

### 14.1 データ整合性ポリシー

- データの正しさはデータモデルで担保する
- フロント補完に依存しない
- 自動生成データには「自動連携フラグ」を持たせる
- 修正・削除は可能とするが、連携取引は警告対象とする
- 集計のためのバッチ前提設計をしない

### 14.2 修正・削除ポリシー

- 実運用を止めない設計にする
- 修正可能な設計を基本とする
- 削除は論理削除も検討する
- 会計連携データは削除制限を検討する

### 14.3 バッチ非依存ポリシー

- 月次締めバッチを前提にしない
- 日次バッチを前提にしない
- リアルタイム入力を即反映させる設計
- 年度切替は明示的に操作する

### 14.4 年度管理ポリシー

- 年度は 9/1 開始
- 年度データは分離可能な構造にする
- 年度繰越は明示的に記録する
- 前年度比較は将来拡張とする

## 16. プロダクト哲学

Windomsは「吹奏楽クラブ運営者の心理的・物理的負担を軽減するOS」である。

- シフト作成者の心理的負担を軽減する
- 情報が分散しない安心感を提供する

- 運営が属人化しない状態を目指す

### 16.1 Windomsという名前

Windoms は  
**Wind Orchestra Management System** の略である。

同時に、

- 吹奏楽（Wind）を支える基盤であり
- 運営の知恵（Wisdom）を構造化するOS

## 27. 資産 > 楽譜仕様（SSOT / DEMO）

### 27.1 目的

- 楽譜リストを、全件表示を起点に検索・追加・編集できる状態にする。
- 既存の楽譜管理スプレッドシート相当の運用を、DEMO上で迷わず再現する。
- `minamigaoka` では、楽譜モジュールの正データを Firestore ルート直下の `scores` コレクションで管理する。
- 既存のローカル楽譜データは初期投入用ソースとして扱い、初回 seed で Firestore に投入する。
- 今回の前提ではクラブごとに Firebase プロジェクトを分けるため、楽譜データに `clubId` や `clubs` 配下のクラブ概念は持ち込まない。

### 27.2 表示・検索

- デフォルトは全件表示とする。
- 検索はリアルタイム絞り込みとし、検索ボタンは置かない。
- 検索対象は `曲名 / 出版社 / 品番 / 備考` とする。
- 入力文字列はスペース区切りでトークン化し、AND条件で絞り込む。
- 大文字小文字は区別しない。
- 文字列比較は可能な範囲で `NFKC` 正規化を行い、全角半角差分を吸収する。
- 文字列比較は可能な範囲で、ひらがな・カタカナ差分も吸収する。
- 当番日誌の活動記録追加モーダルの曲名候補も、この `scores` データと検索正規化を再利用できる構成を基本とする。
- 楽譜一覧・検索は Firestore `scores` のデータを元に動作する。
- Firestore 読込中・空状態・エラー時も、一覧画面で不自然に見えない状態を保つ。

### 27.3 ソート

- テーブルヘッダクリックでソートを切り替える。
- 対象は `No / 曲名`。
- デフォルトは `No 昇順`。
- 同一ヘッダ再クリックで `昇順 ⇄ 降順` をトグルする。
- 現在のソート状態は `▲/▼` などの簡易表示で示す。

### 27.4 編集方針（DEMO）

- 追加・編集は可能、削除は当面行わない。
- 楽譜の検索・閲覧は全員可能とする。
- 楽譜の追加・編集は管理者のみ可能とする。
- 削除は将来のアーカイブ方式で扱う。
- 追加/編集フォーム項目:
  - 必須: `曲名`
  - 任意: `No / 出版社 / 演奏時間 / 品番 / 備考`
- Firestore の保存先は `scores/{docId}` とし、`docId` には既存ローカルデータの `No` を文字列化した値を使ってよい。
- `No` 未入力時は既存最大 `No + 1` を自動採番する（重複は避ける）。
- 入力エラーは該当フィールド直下に表示し、無反応に見える状態を作らない。

### 27.5 一覧表示（GAS画面踏襲）

- 列は `No / 曲名・出版社 / 演奏時間・品番` の3列とする。
- 曲名は太字で1行目、出版社は2行目の補助表示とする。
- 演奏時間は1行目、品番は2行目の補助表示とする。

### 27.6 Firestore 初期投入と再利用方針

- Firestore 初期投入には、現在ローカルで保持している楽譜データをそのまま使う。
- 初期投入は seed スクリプトで行い、同じ `docId` が存在する場合は上書きしてよい。
- seed 実行時は、対象件数 / 成功件数 / 失敗件数が分かるログを出す。
- 今後、当番日誌の活動記録追加モーダルの曲名候補は、この `scores` データを参照する前提とする。
- そのため、楽譜取得ロジックは UI に閉じず、再利用可能な最小限の service / repository 層を持ってよい。

## 28. Docs（資料）モジュール（Memo中心 / Dropbox正）

### 28.1 コンセプト

- 資料ファイルの実体は Dropbox を正（SSOT）とする。
- Windoms は運用メモ（Memo）を中心に扱い、外部リンク（Dropbox等）を参照としてぶら下げる。
- Windomsにしか存在しない長文資料は基本作らず、要点メモ＋参照に寄せる。
- `minamigaoka` では、資料モジュールの正データを Firestore で扱う。
- 資料モジュールはファイル保存機能ではなく、Markdown 本文を持つメモ / 資料ページ管理として扱う。
- Dropbox 等の外部資料は本文中リンクで参照する。
- Storage 連携や実ファイル保存は今回の対象外とする。

### 28.2 データの正本

- Memo本文は Markdown を正本とする。
- 表示時は Markdown を HTML にレンダリングして閲覧する。

### 28.3 UI（v1）

- 一覧（`#/docs`）:
  - Memoの一覧、検索（タイトル/タグ等）、カテゴリ絞り込みを想定する。
  - 一覧・詳細の閲覧は全員可能とする。
- 編集:
  - Markdownをプレーン編集（textarea等）で扱う。
  - 本文中にDropbox等のリンクを自由に貼れる。
  - 追加・編集・削除は管理者のみ可能とする。
- 閲覧:
  - Markdown→HTML整形表示で閲覧する。
  - リンクは外部リンク（新規タブ）で開く。
- 参照リンク（任意）:
  - 本文中リンクに加えて、参照リンク欄（複数/0件OK）を将来設計として許容する。

### 28.4 将来拡張（vNext）

## 19. データモデル（案）

前提（確定事項）:
- 年度は 9/1 開始
- `YYYY-MM-DD` は年度に依存しない絶対日付として保持する
- `scheduleDays` のIDは `YYYY-MM-DD`
- `scheduleDays/{YYYY-MM-DD}` は docId が `YYYY-MM-DD`（フィールドに `id` / `date` は持たない）
- `sessions` は `scheduleDays/{date}/sessions/{sessionId}` のサブコレクション
- `sessions` のIDは自動ID、並び順は `order`（数値）で保証
- `startTime` / `endTime` は `HH:MM` 文字列（JST前提）、基本30分単位（将来拡張可）
- `type` は最小で `normal` / `self` / `event`
- 保護者が不要な日は無い
- `dutyRequirement` を `sessions` に持つ（`duty` / `watch`）
- `duty` は当番シフト割当対象、`watch` は見守り募集対象
- 当番シフト割当（`duty`）と見守り募集（`watch`）は別機能で扱う
- `requiresShift` は当番シフト割当対象を示す旧フラグで、`dutyRequirement` へ移行する
- 互換期間中に `requiresShift` を持つ場合は `dutyRequirement === "duty"` と同義で扱う
- `location` は `sessions` に持てる
- 入力省力化として `scheduleDays.defaultLocation` を任意で持ち、`session.location` が無い場合は `day.defaultLocation` を継承
- 当番者は `sessions.assignees: [uid]` で保持（運用上は原則1名）
- 表示・ics生成省力化のため `assigneeNameSnapshot` を任意で保持可（UIDが正）
- `plannedInstructors` / `plannedSeniors`（予定の講師・先輩）を `sessions` に持てる
- メイン講師がその session に来る予定かどうかを示す `mainInstructorPlanned` を `sessions` に持てる
- 曲マスタは `songMaster/{songId}` として保持する（`name` 必須）
- 日誌は `dayLogs/{YYYY-MM-DD}`（1日1枚）
- 日誌から「活動場所」「顧問」は削除（不要）
- 日誌は備考 `notes` を持つ
- `weather` は任意（ワンタップ想定）
- 当番の割当単位は「家庭（household）」である。
- 保護者ユーザーは家庭に所属する（父母が別ユーザーとして存在し得る）。
- 当番可否アンケートは家庭単位で集約する。
  - 例: 渋谷父が回答した時点で、渋谷家は回答済となり、渋谷母は追加回答不要。
- 兄弟姉妹は、同じ家庭に子どもが複数所属することで表現できる。
- 親子関係は、家庭メンバーの役割（guardian/child）で最低限表現し、個別の親子紐付けは後続フェーズで拡張する。
- 保護者には、家庭内の続柄（父/母/祖母/叔母/その他 など）を持たせる。続柄は users ではなく household のメンバー情報（membership）に紐付ける（家庭内の関係として扱う）。
- 出欠予定は日単位ではなくセッション単位で管理する（`sessions/{sessionId}/rsvps/{uid}`）
- 日誌はスケジュールを参照して入力を省力化する（第17章の通り強制同期・自動上書きはしない）
- 実績側は `actualInstructors` / `actualSeniors` を持ち、来なかったは「予定にいるが実績にいない」差分で表現する（明示フラグは持たない）
- 実績側は `mainInstructorAttendance`（セッション `order` 単位の在席チェック）を持てる
- 捺印実績は `dayLogs.dutyStamps` に保持し、予定当番とは別で扱う

### 19.1 scheduleDays/{date} フィールド例

```json
{
  "defaultLocation": "第1音楽室",
  "notice": "本日16:30に片付け開始です。\n打楽器は搬出前にチェックリストを確認してください。"
}
```

- ドキュメントIDが `YYYY-MM-DD`（フィールドとして保持しない）
- `defaultLocation` は任意
- `notice` は任意（複数行可。Todayでは改行を反映して表示）

### 19.2 sessions/{sessionId} フィールド例

```json
{
  "order": 1,
  "startTime": "09:00",
  "endTime": "12:00",
  "type": "normal",
  "eventName": "",
  "dutyRequirement": "duty",
  "requiresShift": true,
  "location": "第1音楽室",
  "assigneeFamilyId": "family_123",
  "assignees": ["uid_123"],
  "assigneeNameSnapshot": "山田花子",
  "note": "午後は片付けを優先",
  "mainInstructorPlanned": true,
  "plannedInstructors": ["講師A"],
  "plannedSeniors": ["先輩B"]
}
```

- `sessionId` は自動ID
- 並び順は `order` で制御
- `dutyRequirement` は `duty` / `watch`
- `requiresShift` は旧フラグ（互換）。`dutyRequirement === "duty"` と同義
- `type === "event"` のときは `eventName` を event セッション名として扱ってよい
- `assigneeFamilyId` は当番 / 見守り担当の `family` を指せる
- `note` は任意（セッション単位メモ）
- `mainInstructorPlanned` はメイン講師の出席予定を表す。`true / false / null(or未設定)` を許容する
- `location` 継承の優先順位:
1. `session.location`
2. `scheduleDays.defaultLocation`
3. 未設定（空欄）

### 19.3 dayLogs/{date} 最小フィールド例

```json
{
  "notes": "備考メモ",
  "weather": "晴れ",
  "activities": [
    {
      "startTime": "09:05",
      "type": "基礎練習",
      "title": "ロングトーン",
      "songIds": ["song_001", "song_002"]
    }
  ],
  "dutyStamps": {
    "1": {
      "stampedByUid": "demo_writer",
      "stampedByName": "日誌記入者",
      "stampedAt": "2026-02-15T00:12:34.000Z"
    }
  },
  "actualInstructors": ["講師A"],
  "actualSeniors": ["先輩B"],
  "mainInstructorAttendance": {
    "1": true,
    "2": false
  }
}
```

- ドキュメントIDが `YYYY-MM-DD`（フィールドとして保持しない）
- `activities` は活動ログ（開始時刻ベース）
- `activities` は `startTime` 昇順で表示・扱う
- 同一 `startTime` がある場合は登録順で扱う
- `activities.songIds` は任意（0件以上、複数曲を許容）
- `actualInstructors` / `actualSeniors` は `string[]` として保持する
- `mainInstructorAttendance` はセッション `order` をキーにした在席実績（boolean）を保持する
- 実績名の重複は許容する
- 「来なかった」は予定と実績の差分で扱う
- `dutyStamps` はセッション枠ごとの捺印実績を保持する
- `dutyStamps` のキーはセッションの `order` を文字列化した値を使用する
- `dutyStamps.{order}` は `stampedByUid` / `stampedByName` / `stampedAt` を持つ

### 19.4 sessions/{sessionId}/rsvps/{uid} フィールド例（出欠予定）

```json
{
  "status": "yes",
  "comment": "部活後に通院予定",
  "updatedAt": "serverTimestamp",
  "displayNameSnapshot": "田中太郎"
}
```

- 管理単位はセッション単位（AM/PM など枠ごと）
- `status` は `yes` / `maybe` / `no` / `unknown`（`unknown` は未回答）
- カード表示では `unknown` を集計・表示しない
- Today の出欠モーダルは参照専用（編集不可）
- Today の出欠モーダルでは `unknown` を `-` 表示する
- 日誌の出欠モーダルは編集可能（`yes` / `maybe` / `no` の排他選択）
- 日誌の出欠モーダル保存時は未選択（`unknown`）を許可しない

### 19.5 計画→実績の関係（実装方針）

- 日誌入力時にスケジュール情報を参照して初期表示に利用する
- 計画と実績は強制同期しない
- 自動上書きはしない

### 19.6 ドメイン/ホスティング/テナント設計（SSOT）

- Windoms のWeb公開ドメインは `windoms.club` を前提とする。
- クラブごとにサブドメインを割り当てる。
  - 例: `minamigaoka.windoms.club`
- サブドメイン運用は将来のクラブ追加を前提とする。
- ホスティング基盤は Firebase Hosting を採用する。
- Firebase Hosting は multi-site を採用し、クラブごとに Hosting サイトを分離する。
  - 各クラブのサブドメインを、対応する Hosting サイトに紐付ける。
- Firestore は当面、単一 Firebase プロジェクトで共通運用する。
- 全データは必ず `clubs/{clubId}/...` 配下に格納する（クラブ分離の第一キー）。
- 原則として `サブドメイン名 = clubId` とする。
  - 例: `minamigaoka.windoms.club` の `clubId` は `minamigaoka`
- 将来、特定クラブが個別最適で別物化した場合のみ、そのクラブに限って別 Firebase プロジェクトへ分離可能とする。
  - ただし初期構成は単一 Firebase プロジェクトで開始する。

#### 19.6.0 ドメインの役割分担

- Windoms のドメインは次の役割で運用する。
- `windoms.club`
  - サービス紹介ページ
  - ドキュメント / 導入案内
  - 将来のLP
- `demo.windoms.club`
  - Windoms デモ環境
- `{clubId}.windoms.club`
  - 各クラブの Windoms インスタンス
- 例:
  - `minamigaoka.windoms.club`
  - `sakuragaoka.windoms.club`
- サブドメイン = `clubId` とする。

#### 19.6.1 Firebase Hosting multi-site の上限

- 1 Firebase プロジェクトあたりの Hosting サイトは最大36サイトまでとする。
- 10クラブ程度の運用規模では上限上の問題は発生しない前提とする。
- 36サイトを超える見込みが出た時点で、Firebase プロジェクト分割を検討する。

#### 19.6.2 サブドメインと Hosting サイトの対応

- 原則として `サブドメイン名 = clubId` とする。
- 1クラブ = 1 Hosting サイトで運用する。
- 対応例:
  - `demo.windoms.club` -> Hosting site `windoms-demo`
  - `minamigaoka.windoms.club` -> Hosting site `windoms-minamigaoka`（例）

#### 19.6.2.1 siteId / target 命名規約

- Firebase Hosting multi-site では次の命名ルールを採用する。
- `siteId`
  - `windoms-{clubId}`
- 例:
  - `windoms-demo`
  - `windoms-minamigaoka`
- `target`（ローカル deploy 名）
  - `clubId` または `demo`
- 例:
  - `firebase deploy --only hosting:demo`
- `siteId` は Firebase 全体で一意である必要がある。

#### 19.6.3 DNS 設定ルール

- サブドメインは CNAME で `SITE_ID.web.app` に向ける。
- 例: `demo.windoms.club` は `windoms-demo.web.app` に向ける。
- 同一ホスト名に A/AAAA レコードと CNAME レコードを混在させない（競合回避）。
- DNS 伝播および SSL 証明書の発行反映には最大24時間かかることがある。

#### 19.6.3.1 DNS トラブルシュート

- DNS が正しく設定されているかは次のコマンドで確認する。
- `nslookup -type=cname demo.windoms.club`
- 期待結果:
  - `demo.windoms.club canonical name = windoms-demo.web.app`
- 注意:
  - 同じホストに A / AAAA と CNAME を共存させない
  - DNS 反映には数分〜数時間かかる

#### 19.6.3.2 SSL 証明書反映とキャッシュ

- Firebase Hosting でカスタムドメインを追加すると、SSL 証明書の発行に最大24時間かかる場合がある。
- 証明書が発行された直後はブラウザキャッシュにより「保護されていない通信」と表示されることがある。
- 確認方法:
  - シークレットウィンドウでアクセス
  - 強制リロード（Ctrl + Shift + R）

#### 19.6.4 追加クラブを作る手順（Hosting側）

- Firebase CLI で `hosting:sites:create` を実行し、クラブ用サイトを作成する。
- `firebase target:apply hosting` でターゲット名と `SITE_ID` を紐付ける。
- `firebase.json` に `hosting` エントリを追加し、クラブ別に `public` ディレクトリを分ける。
  - 例: `demo` は `demo/dist`
  - 将来クラブも `clubName/dist` のように分離する。
- デプロイは `firebase deploy --only hosting:ターゲット名` で個別実行する。
- 一括デプロイ運用を行う場合は、意図しないクラブまで同時反映されるリスクに注意する（基本は個別）。

#### 19.6.5 デプロイ運用（ローカル）

- `demo` は Vite のビルド出力 `demo/dist` を Hosting 対象とする。
- ローカル運用の基本フロー:
  - `demo` ディレクトリで `npm run build` を実行する。
  - リポジトリルートで `firebase deploy --only hosting:demo` を実行する。
- 実際に配信されるディレクトリは `firebase.json` の `public` 設定に依存するため、変更時は必ず整合を確認する。

#### 19.6.6 GitHub 自動デプロイについて

- Firebase Hosting は GitHub Actions による自動デプロイも可能だが、Windoms では当面手動 deploy を採用する。
- 理由:
  - デモ段階では手動の方が安全
  - multi-site deploy の管理を簡単にするため
- 基本 deploy:
  - `cd demo`
  - `npm run build`
  - `firebase deploy --only hosting:demo`

## 21. 設計思想の追記（Session / Event / Task / Today / Action Bar）

### 21.1 データモデル設計思想（Session / Event / Task）

- Session：日時を持つ活動単位（Today/日誌の起点）
- Event：意味のまとまり（コンサート/コンクール/合宿等）。時間はSession側に持たせる考え方（Event自体は日時必須ではない）
- Task：実行単位（TODO）

### 21.2 EventとSessionの関係

- Eventは0個以上のSessionを持てる
- SessionはEventを0 or 1持つ
- N対Nは不要（当面やらない）
- 例：合宿はEvent1つに対して複数Session（各日の練習枠）
- 例：本番に向けた事前練習3回も同じEventに紐付けて見られる

### 21.3 Taskの紐付け（XORルール）

- Taskは紐付け先を最大1つ（none / sessionId / eventId）
- 同時にsessionIdとeventIdの両方は持たない
- Eventに紐付いたTaskは、関連Session側でも「表示」は可能（データ重複はしない）

### 21.4 Today画面の役割（Notice含む）

- Todayは運営の現在地（起点画面）
- Noticeは「手動作成型」と「自動生成型」を想定
- 通知ログではなく「今、知るべきこと」を表示する層

### 21.5 Action Bar（ステータスバー）

- Todayヘッダー右側にステータスアイコン（🔔/✅/📅）を置く
- Action Bar（ステータスバー）はWindoms共通ヘッダーとして全画面に表示する
- バッジで件数を示し、タップで各導線（受信箱/画面）を開く
- ハンバーガーは最右固定（メニューは既存のオーバーレイ方式を維持）

### 21.5.1 ヘッダー右上UIの役割

- DEMOバッジの常時表示は行わない（スマホ表示でヘッダーが限界のため）。
- DEMOであることの表示は、将来実装する設定画面（`#/settings`）に集約する。
  - 表示例: `環境: DEMO（データは仮）`
- 設定画面が未実装の間は、画面上にDEMO表示が無い状態を許容する。
- 🔔通知はシステム主導のアクションを扱う。
- ✅TODOはユーザー主導のタスク管理を扱う。
- 常設メニューに存在しない機能でも、状況に応じてヘッダーにアイコン導線を出してよい。
- 「次の当番」は保護者向け表示とし、部員 / 先輩 / 先生には表示しない。
- 「次の当番」はログイン中ユーザー本人ではなく、そのユーザーが所属する Family 単位で判定する。
- 表示対象は、ログイン中ユーザーが所属する Family に割り当てられた今後の最も近い当番セッションとする。
- 次回当番が無い場合は `-` を表示する。
- 次回当番が土日PMのセッションである場合のみ、お弁当アイコンを表示する。
- お弁当アイコンはお弁当モジュールトップ（`/lunch`）への導線とする。
- 今回は `次の当番` とお弁当アイコンのみを対象とし、TODO / 通知など他のステータスバー要素は変更対象外とする。
- 「次の当番」表示はレスポンシブで切替する（`768px以上=1行`、`767px以下=label/date/timeの3行`）。

### 21.5.2 通知パネル仕様

- ステータスバー通知は、特定モジュール専用ではなく Windoms 全体で共通利用する横断通知基盤として扱う。
- v1 では Firestore 正式化と最小運用を目的とし、右上通知アイコン / 通知センターは Firestore 上の通知データを正として表示する。
- v1 の通知生成元第一弾は、管理者専用の手動通知モジュール `システム通知` とする。
- 将来的には、購入依頼・立替・お弁当・会費管理・TODO などの状態変化通知を、この共通通知基盤に接続できる前提とするが、今回は実装しない。
- 通知履歴は Windoms 全体の横断履歴とし、手動通知と将来の自動通知の両方を同じ履歴基盤へ記録する。
- v1 ではまず手動通知で履歴利用を開始し、自動通知を追加する際も同じ履歴基盤へ記録する前提とする。
- 通知は `users/{uid}/notifications/{notificationId}` に保存する。
- 共通コレクション方式ではなく、対象ユーザーごとの配下へ通知を複製配布する方式を採用する。
- 理由は、ユーザー別取得、バッジ件数集計、Rules 設計、通知センター UI 実装をシンプルに保つためである。
- 受信者ごとの状態管理とは別に、通知が発生した事実を記録する共通履歴親レコードを `notificationHistory/{notificationHistoryId}` に保存する。
- `users/{uid}/notifications` は受信者ごとの状態管理、`notificationHistory` は通知が発生した事実の記録、という責務分担にする。
- 通知状態は、閲覧状態と対応状態の 2 軸で持つ。
- 閲覧状態は `isRead: false | true` と `readAt: Timestamp | null` で管理する。
- 対応状態は `status: active | resolved` と `resolvedAt: Timestamp | null` で管理する。
- 既読と対応済は別概念として扱う。
- 通知センターのタブ文言は「未対応」「対応済」とする。
- UI 上の「未対応」は内部値 `status=active` に対応し、UI 上の「対応済」は内部値 `status=resolved` に対応する。
- 通知カードの操作ボタン文言は `対応済にする` とする。
- 通知センターは「未対応」「対応済」の 2 タブ構成とする。
- 未対応タブには `status=active` を表示し、対応済タブには `status=resolved` を表示する。
- 通知カードをクリックした時点で、未読通知は既読化する。
- 既読化では `status` は変更しない。
- `対応済にする` 押下時は `status=resolved`, `resolvedAt=now` に更新する。
- 対応済操作時に未読だった場合は、あわせて `isRead=true`, `readAt=now` にしてよい。
- 対応済詳細モーダルには `未対応に戻す` を表示してよく、押下時は `status=active`, `resolvedAt=null` に戻す。
- `未対応に戻す` 操作では `isRead` / `readAt` は維持する。
- 一覧を開いただけでは既読化しない。
- 右上通知アイコンのバッジ件数は未読件数ではなく `status=active` 件数とする。
- 読んでも未対応ならバッジは残り、対応済になった時点でバッジから消える。
- v1 の通知データ項目は、少なくとも `type`, `title`, `body`, `isRead`, `readAt`, `status`, `resolvedAt`, `createdAt`, `sourceModule`, `sourceEvent`, `sourceRecordId`, `linkType`, `linkUrl`, `senderUid`, `senderName` を持つ。
- v1 の通知履歴親レコード項目は、少なくとも `kind`, `sourceModule`, `sourceEvent`, `title`, `body`, `targetType`, `targetSummary`, `targetUserIds`, `recipientCount`, `createdAt`, `createdByUid`, `createdByName`, `isCancelable`, `sourceRecordId`, `canceledAt`, `canceledByUid`, `canceledByName` を持つ。
- 手動通知は `kind=manual`、将来の自動通知は `kind=auto` とする。
- 手動通知のみ `isCancelable=true` を基本とし、自動通知履歴は参照専用とする。
- 管理者は、タイトル、本文、通知対象範囲を指定して通知を送信できる。
- 送信時は対象ユーザーごとに `users/{uid}/notifications` へ通知ドキュメントを作成する。
- 手動通知送信時は、各ユーザー配下通知の作成に加えて、`notificationHistory` にも 1 件記録する。
- システム通知の送信元管理記録は `systemNotifications/{systemNotificationId}` に保持してよいが、通知履歴一覧は `notificationHistory` を正とする。
- v1 の送信対象範囲は `全員` `管理者のみ` `保護者のみ` `子どものみ` `個別ユーザー選択` とする。
- 個別ユーザー選択時の対象選択 UI は、複数候補を一覧表示する `選択リスト` 方式を許容する。
- 各行の先頭にチェックボックスを表示し、必須入力エラーは対象ユーザー欄のフィールド単位で表示する。
- 個別ユーザー選択時に 1 件も選ばれていない場合は送信できない。
- 通知対象ユーザー選択肢は、Windoms 内の既存の基本表示順を再利用する。
- 大きな括りは `部員 → 保護者 → 先生` の順を基本とし、同一括り内の並びも既存のメンバー表示順に揃える。
- 個別ユーザー選択リストは、各候補を 1 行で安定表示することを優先する。
- 各行は `チェックボックス / 氏名 / 区分` の横並びを基本とする。
- 氏名は通常の横書きで表示し、1 文字ずつの縦落ちや過剰な折り返しを許容しない。
- スクロール領域を持つ場合でも、各行の高さはコンパクトに保つ。
- チェックボックス列は固定幅寄り、氏名列は主列、区分列は補助列として扱う。
- 通知センター一覧には本文を出さず、タイトルのみを表示する。
- 一覧で必要なら作成日時や未読状態の見た目は保持してよいが、本文確認は一覧ではなく詳細モーダルを主導線とする。
- 通知カードクリックで通知詳細モーダルを開き、詳細モーダルには少なくとも `タイトル` `本文全文` `作成日時` を表示する。
- 必要に応じて送信者名も表示してよい。
- 通知詳細モーダルを開いた時点で、未読通知は既読化する。
- 一覧を開いただけでは既読化しない。
- `対応済にする` 操作は通知詳細モーダルから行える構成を基本とする。
- 通知履歴一覧は管理者向けに月単位で閲覧できるようにし、表示中の年月を `前月 / 翌月` で切り替える。
- 履歴一覧の絞り込み文言は `すべて` `手動` `自動` とする。
- 履歴一覧カードは `タイトル` `本文` `送信時刻` `人数` を中心にシンプルに表示し、詳細情報は詳細モーダルで確認する。
- 履歴一覧カードクリックで通知履歴詳細モーダルを開き、少なくとも `タイトル` `本文全文` `送信日時` `種別（手動 / 自動）` `由来モジュール` `対象範囲` `人数` を表示する。
- 手動通知履歴のみ、詳細モーダルに削除ボタンを表示する。
- 手動通知履歴の削除は単なる履歴削除ではなく誤送信時の `通知取消` として扱い、履歴親レコードと、その履歴から配布された全ユーザー分の通知を削除する。
- `通知取消` の対象は手動通知のみとし、自動通知履歴は削除不可とする。
- 将来的に、購入依頼したレコードが担当者によって購入済みになった場合に依頼者本人へ通知するなど、各モジュールの状態変化通知を同一基盤へ載せられる構成を想定する。
- v1 で扱わないものは、push 通知、通知設定、一括既読、一括対応済、配信予約、高度な絞り込み条件、他モジュールからの自動通知生成、履歴一覧の複雑な検索、年単位集計や高度な分析表示、通知詳細のページ遷移化とする。

### 21.5.3 TODO導線仕様（受信箱）

- ヘッダーの✅TODOは「自分の受信箱」導線として扱う。
- 受信箱は `assigneeUid == 自分` かつ `completed == false` のTODOを表示する。
- 受信箱の並びは「期限が近い順（dueDate昇順）」を基本とする。
- 全体棚卸し・アサイン調整はメニューのTODOページで行う（常駐パネル前提にしない）。

### 21.5.4 UIレイヤー設計（z-index）

- 重なり順は以下を基準とする。
- 常駐パネル: 1000
- オーバーレイ: 2000
- モーダル本体: 2100
- 通知/TODO/DEV等の常駐UIより、確認モーダルを常に前面表示する。

### 21.5.5 ボタン表記規約（v1）

- 一覧画面の「追加」導線は、基本を `＋ 追加` とする。
- 横幅が厳しい場合のみ `＋`（アイコンのみ）を許可する。
- `＋`（アイコンのみ）の場合は `title` と `aria-label` を `追加` に統一する。
- モーダル/フォームの主要ボタンは以下に統一する。
  - 追加系の確定: `追加`
  - 保存系の確定: `保存`
  - 決定系（意思決定）: `確定`
  - 取り消し: `キャンセル`
- 削除など危険操作のボタン表記は `削除` に統一する。
- モーダル/メニューを閉じる操作は `閉じる` に統一する。
- `×` アイコンを使う場合も `title` と `aria-label` は `閉じる` に統一する。
- 英語表記（`Save` / `Cancel` / `Add` / `Create` など）が混在する場合は日本語に統一する。

### 21.6 表示ルール（Taskの見せ方）

- Sessionに紐付くTaskは、そのSession詳細画面に表示される。
- Eventに紐付くTaskは、Event詳細画面に表示される。
- Event配下のSession画面では、Event由来のTaskも参照表示できる（ただしデータは重複保存しない）。
- Session由来のTaskとEvent由来のTaskは、表示上区別できるようにする（例：バッジ表示など）。
- Taskの保存先は常に一意であり、表示のためにコピーを作らない。

## 22. スケジュール機能のモジュール分離

### 22.1 カレンダー（全員向け）

- クラブ共有予定の閲覧画面
- 個人予定は自分のみ編集可能（他人非公開）
- クラブ予定は全員閲覧可能
- 「出来上がった世界を見る」モジュール

### 22.2 活動予定（管理者向け）

活動予定は「月を作る場所」である。  
- DEMO画面のタイトル表示は、役割が伝わるよう `シフト作成`（旧: 活動予定）を使用してよい。

- 活動予定（activity-plan）の作成対象月は、原則「翌月」とする。
  - 例: 現在が 2026-02 の場合、操作対象は 2026-03。
- DEMO段階では「翌月固定」でよい（対象月の切替UIは後回し）。

### 22.2.1 活動予定（MonthlyPlan）の最小データ

活動予定は「月次の作業台」であり、月ごとに1つ存在する。

- monthKey: "YYYY-MM"（例: "2026-02"）
- status: NOT_STARTED | SESSIONS_SET | SURVEY_OPEN | SURVEY_CLOSED | AI_DRAFTED | SHIFT_CONFIRMED | NOTIFIED
- updatedAt: ISO文字列またはタイムスタンプ
- publishedAt?: ISO文字列またはタイムスタンプ（通知済みになった時点でセット）

DEMO段階では、MonthlyPlanの永続化は仮実装でよい（ローカル状態でも可）。

### 22.2.2 活動予定（月次）画面の表示方針

活動予定の月次画面は「格子カレンダー」ではなく、帳票（リスト）形式で表示する。
1日を1行として縦に日付が並び、各行にその日のセッション枠・当番・先生予定・外部指導者・備考を表示する。
見た目は現行の月次活動予定ドキュメント（PDF）に近い形式を目指す。
- 当番は独立列ではなく、当番が必要なセッション（通常練習/イベント等）のセル内に表示する。
- 自主練は見守り表示を行わない（見守りは別モジュールで扱う）。
- 「基本プリセットを入れる」は `NOT_STARTED` のときのみ表示する。
- セッション追加は `NOT_STARTED` のときのみ可能とし、月次表の空セル（AM/PM/夜）から行う。
- セッション削除（×）とプリセット投入も `NOT_STARTED` のときのみ可能とする。
- `NOT_STARTED` では下書きを作成し、確認後に「練習日を決定する」で `SESSIONS_SET` に進める。
- 先生予定・備考は全ステータスで編集可能とする。
- プリセット投入時は、土日でセッションが入る日の先生予定に `9:00-15:00` を初期投入する（既存値は上書きしない）。
- 外部指導者・先輩等の随時追記情報は、この作成フェーズ画面では扱わない（後続フェーズ/別画面で扱う）。

月次フローは以下の状態で進行する：

1. NOT_STARTED
2. SESSIONS_SET
3. SURVEY_OPEN
4. SURVEY_CLOSED
5. AI_DRAFTED
6. SHIFT_CONFIRMED
7. NOTIFIED

- シフトは独立モジュールではなく、活動予定の一工程とする
- NOTIFIED後も閲覧・PDF出力は可能だが、活動予定・当番シフト・先生予定・備考は変更不可とする。
- 状態は月次儀式の到達点であり、変更時に巻き戻さない

### 22.2.3 アンケートの位置づけ（当番可否/参加可否）

- Windomsには「アンケート」が2種類ある。
  - 当番可否アンケート（工程内）: MonthlyStatus の SURVEY_OPEN / SURVEY_CLOSED は「当番可否」を集める工程である。ここで集めた結果を元に、当番仮割当（AI_DRAFTED）→当番確定へ進む。
  - 練習参加可否（工程外・常設）: 練習日と当番シフトが確定し、PUBLISHED（通知済）になったタイミングで「参加可否の入力」が可能になる。参加可否はその後も随時更新できる想定であり、MonthlyStatus の工程とは独立した常設機能として扱う。参加可否の入力画面/導線は後続フェーズで設計する。
- PUBLISHED（通知済）到達時に「練習予定・当番シフトの確定通知」と「全員が確認できる共有手段の提供（URL/PDF等）」と「参加可否入力の解放」が同時に成立する。

### 22.2.4 当番可否アンケート仕様（工程内）

- 対象: 保護者全員
- 質問: 当番が必要なセッションを縦に並べ、各セッションに対して ◯ / △ / × で回答する
- 期限: 送信時に期限を設定でき、デフォルトは「3日後」
- 実行可能ステータス: SESSIONS_SET（練習日決定）でのみ「アンケート送信」が可能。それ以外では送信UIを出さない
- Windoms内で実施（外部フォームは使わない）
- SURVEY_CLOSED 到達後、「AIで当番を仮割当」を実行できる。
- 仮割当の実行により MonthlyStatus は AI_DRAFTED へ遷移する。
- AI_DRAFTED の期間のみ、当番（duty）の手修正が可能とする（他ステータスでは不可）。
- 見守り（watch）は本モジュールでは割当・編集しない。
- SURVEY_CLOSED では「AIで当番を仮割当」だけでなく、「AIを使わずに当番編集へ進む（AI_DRAFTEDへ遷移）」導線も持つ（AI不調時等の緊急回避）。
- AI_DRAFTED の当番編集は「候補から選択する」方式とし、自由入力はしない。
- 当番可否（◯△×）の結果は、当番編集時に参照できる形にする（入力UI/連動は後続フェーズで実装し、DEMOでは未実装表示でもよい）。
- AI_DRAFTED 以降、月内の各家庭の当番割当数を確認できる（モーダル等）。
- 手修正による割当変更は、この集計に即時反映される。
- AI_DRAFTED の次ステータスとして SHIFT_CONFIRMED を持つ。
- AI_DRAFTED では「シフトを確定する」操作を実行でき、確認後に SHIFT_CONFIRMED へ遷移する。
- SHIFT_CONFIRMED では当番編集をロックし、閲覧のみ可能とする。
- SHIFT_CONFIRMED の編集ロックはUI表示だけでなく、ロジック上でも当番編集不可を保証する。
- SHIFT_CONFIRMED では当番回数確認モーダル等の閲覧機能は利用できる。
- SHIFT_CONFIRMED では当番編集に加えて、先生予定・備考の編集もロックする。
- SHIFT_CONFIRMED では PDF出力（複数回可）と「全メンバーに通知する」操作のみ実行できる。
- 「全メンバーに通知する」実行後は NOTIFIED に遷移し、通知ボタンは再表示しない。
- NOTIFIED は活動予定と当番シフトの通知が完了した状態を示し、出欠予定入力を解放する。

## 通知方針

- 自動LINE通知は必須機能としない（将来オプション）
- Windomsはアプリ内で完結可能な設計とする
- 通知は画面上の状態遷移（SHIFT_CONFIRMED → NOTIFIED）で管理する

## LINE併用とパーマリンク方針

### 基本思想

- Windomsはすべての公式情報のSingle Source of Truth（SSOT）である
- LINEグループ等の外部チャネルは併用するが、共有するのはWindoms内のリンクとする
- 自動LINE通知は必須とせず、手動共有を前提とする

### パーマリンクの原則

- 一般メンバーが閲覧する公開面はパーマリンクを持つ
  - カレンダー
  - 出欠
  - 当番可否アンケート（`/shift-survey?month=YYYY-MM`）
  - イベント（`/events`、詳細 `events/:eventId`）
  - TODO（将来）
  - 資料（将来）
- 運営専用の作業面（例: 活動予定モジュール）は必須ではない

### URL設計ルール

- 安定したキー（dateKey / monthKey / itemId 等）で参照する
- URLは可能な限り変更しない設計とする
- 一覧表示と個別詳細の両方に直リンク可能な構造を推奨する

### リンクコピーUI方針

- ブラウザURL欄のコピーで代替可能なため、リンクコピーボタンは必須ではない
- 共有頻度が高いモジュールでは、実装時にリンクコピーボタン追加を検討する
- ボタンの要否は各モジュール設計時に判断する

### 運用イメージ

- LINEには説明文＋Windomsリンクを共有する
- 当番可否アンケートURL（`/shift-survey?month=YYYY-MM`）は運営が手動共有して回答を促す
- 情報の更新は必ずWindoms側で行う
- 「LINEは導線、Windomsが本体」という役割分離を維持する
- 当番可否アンケートは「やること」としてTODOに集約し、新しい右上アイコンは追加しない

### 22.3 見守り（自主練）

- 自主練は通常セッションと同様に1セッション
- `dutyRequirement: "watch"` を持つセッションが対象
- 月初に割り当てない
- 随時挙手で1枠1人を募集
- 未確定 / 確定状態を可視化
- 日誌上では当番と同様に捺印対象となり得る

### 22.4 リンク集（外部ポータル）

- リンク集は Windoms 外部サービスへのポータルとして扱う。
- Windoms内部モジュール（会計・シフト・カレンダー・楽譜等）に吸収される機能はリンク集に含めない。
- 外部リンクは別タブで開く。
- 一覧の閲覧と外部リンク遷移は全員可能とする。
- 追加・編集・削除は管理者のみ可能とする。
- 一部リンクはロール（役員）に応じて表示制御する。
- DEMOはレベル2（`type` / `role`）で実装し、将来はOGP/favicon自動取得を含むレベル3へ拡張する。
- 役員は同一画面でリンクの追加 / 編集 / 削除を行える（独立編集画面は作らない）。
- `minamigaoka` では、リンク集モジュールの正データを Firestore で扱う。
- 既存サンプルデータは、本番想定の初期データとして seed で Firestore に投入する。
- 画面側は Firestore を正として表示 / 追加 / 編集 / 削除を行う。

### 22.5 メンバー

- メンバーは部員 / 保護者 / 先生を含む。
- 保護者⇄部員の紐づけを持ち、一覧では件数リンク（保護者: N / 子ども: N）から関係を閲覧できる。
- 紐づけの追加 / 解除 / 編集は将来、役員等の権限で制御する想定とし、今回は閲覧導線のみを提供する。
- 部員の学年は `enrollmentYear`（入学年度）から年度計算で自動算出し、4月で自動切替する。
- パートはまず正式名称で表示し、将来は略称表示へ拡張可能とする。
- 一般向けメンバーモジュールでも、フリガナは `familyNameKana` と `givenNameKana` を連結したフルネーム表示とする。片方のみある場合は、その値だけを表示してよい。
- DEMOのメンバーモジュール表示は例外として、部員はフルネーム＋ふりがな表示、保護者は `姓（父/母）`、先生は `井野 勝彦` とする。

### 22.6 イベント（行事管理）

- イベントは行事管理の箱として扱い、セッションへの紐づけは0件以上を許容する。
- イベント一覧は `/#/events`、イベント詳細は `/#/events/:eventId` をパーマリンクとする。
- イベントは `state`（`active` / `done`）で管理し、一覧区分は日付自動判定ではなく状態で行う。
- イベントは `eventSortDate` を持ち、一覧の並び順と完了タブの年度判定キーとして使う。
- `eventSortDate` はイベント本体の代表日として保持し、セッション紐付けの有無に関わらず残す。
- 一覧は「進行中 / 完了」タブで切り替える。
- 完了タブには年度フィルタ（9/1開始）を持つ。
- イベント詳細から「このイベントのTODO一覧」を確認できる。
- イベント詳細で `完了にする / 進行中に戻す` を切り替えできる。
- イベント詳細から「紐付けセッション一覧」を開き、既存セッション表示UIと統一感を持たせる。
- イベントの追加 / 編集 / 削除は権限ユーザーのみ実行できる（閲覧ユーザーは一覧・詳細のみ）。
- 一覧ヘッダー右上の `＋` で追加し、カード右上の編集/削除アイコンで更新する。
- `minamigaoka` では、イベント本体の正データを Firestore で扱う。
- イベント一覧 / 追加 / 編集 / 削除 / 完了切替は、同じ Firestore イベントデータを参照して永続化する。
- TODO の関連表示やイベント選択UIなど、イベント本体を参照する主要UIもこの Firestore イベントデータを参照する。
- 今回の段階では高度なリアルタイム最適化より、永続化と参照一元化を優先する。
- イベント一覧 / イベント詳細の閲覧は全員可能とする。
- イベントの追加 / 編集 / 削除 / 完了切替は管理者のみとする。
- イベント詳細でのセッション紐付け / 解除も管理者のみとする。
- Today / カレンダーなどからイベント詳細へ入る導線は全員に表示してよい。
- 非管理者には管理操作を基本的に非表示とし、「押せないが見えている」状態は避ける。
- イベント種別は `コンクール / 演奏会 / 合同練習 / その他` の固定セレクトとする（自由入力なし）。
- 種別バッジは色分けせず、同一スタイルでテキスト表示のみとする。
- イベントカードの編集/削除アイコンは右下に配置し、右上は状態/種別バッジの表示専用エリアとする。
- イベントに紐づけ可能なセッションは当面 `type=event` のみとする。
- イベントはセッション 0 件でも単独で存在できる。
- イベント詳細を、イベント本体確認・関連TODO確認・紐付けセッション管理の中心画面とする。
- セッション紐付けはイベント詳細から手動で行う。
- 紐付け候補は当面 `type=event` のセッションに限る。
- 1つのセッションは同時に複数イベントへは属さない。
- イベント詳細の紐付けセッション一覧では、複数セッションを一覧表示でき、1件ずつ追加 / 解除できる。
- README に自動で確定紐付けする仕様は持たず、明示的に紐付けたものだけを紐付け済みとして扱う。
- Today / カレンダーなど、イベント表示がある主要UIからイベント詳細へ遷移できる導線を持てるようにする。
- 「解除」は削除ではなく、イベントとセッションの紐づけ解除を意味する。

## 23. ルーティング設計方針（DEMO段階）

### 23.1 モジュール単位の独立ルート

Windomsでは、モジュールとして存在が確定した時点で独立ルートを与える。

例:
- /today
- /activity-plan
- /calendar
- /journal

独立ルートは、そのモジュールが今後仕様を詰めていく前提で存在することを意味する。

### 23.2 today?view= の扱い

/today?view=xxx は、以下用途でのみ使用する：
- 仕様未確定モジュールの仮表示
- DEMO中の一時的な画面確認
- 構造検討中のUIの暫定置き場

正式モジュールに昇格したものは、today?view= では扱わない。

### 23.3 昇格ルール

- 機能として存在が確定
- 今後継続して仕様を詰める対象
- ナビゲーションに正式表示する

これらを満たした時点で独立ルートを与える。

### 23.4 現在の適用状況（DEMO）

- activity-plan は正式モジュールとして独立ルートを持つ
- その他未確定モジュールは today?view= のままとする
- トップ（`/` / `#` / `#/`）は `location.replace("#/today")` で Today へリダイレクトする（履歴を残さない）

## 24. TODOモジュール仕様（SSOT）

本章を、Windoms DEMOにおけるTODO仕様のSingle Source of Truth（SSOT）とする。

### 24.1 役割（2階建て）

- 上部TODOアイコン（✅）: 自分の受信箱（自分の未完了TODOを素早く確認）
- メニュー「TODO」ページ: 全体棚卸し（未アサイン優先表示）＋アサイン調整（引き継ぎ含む）
- `minamigaoka` では、TODO の正データを Firestore で扱う。
- 主要な参照UI（TODOページ本体 / 上部タスクバー受信箱 / セッション関連UI / イベント関連UI）は、同じ Firestore TODO データを参照する。
- 今回の段階では、リアルタイム購読の高度化よりも、永続化と参照元の一元化を優先する。
- TODO は `共有TODO` と `個人TODO` の 2 種類を扱う。
- `個人TODO` は全メンバーが作成でき、作成者本人にしか見えない。担当は持たず、セッション / イベント紐付けも持たない。
- `共有TODO` は全体共有ではなく、共有範囲を持つ。担当を持て、セッションまたはイベントに紐付けできる。
- TODO はタイトルに加えて任意のメモを持てる。タイトルは必須、メモは任意とし、`共有TODO` / `個人TODO` の両方で同じ考え方で扱う。
- メモ欄は複数行の自由入力とし、Markdown 記法はサポートしない。
- TODO メモの表示では改行を自然に表示し、`http://` または `https://` の URL 文字列は自動リンク化してよい。それ以外の装飾解釈はしない。
- `イベント` / `購入依頼` / `立替` / `お弁当` / `会計` の `memo` / `note` / `comment` / `description` / `備考` など自由記述欄も、保存時はプレーンテキストのまま保持し、表示時のみ同じルールを適用してよい。
- これらの自由記述欄は、改行保持・`http://` / `https://` の自動リンク化を行ってよいが、Markdown や HTML は解釈しない。
- `共有TODO` の共有範囲は、既存ロール体系に合わせて `parent` / `officer` / `child` を使う。
- `共有TODO` は、その共有範囲で閲覧可能なメンバーにのみ表示する。
- 上部TODOアイコン（受信箱）には「自分担当の未完了共有TODO」と「自分の未完了個人TODO」の両方を表示する。

### 24.2 TODOデータモデル（DEMO最小）

- `kind: 'shared' | 'private'`
  - `private = 個人TODO`
  - `shared = 共有TODO`
- `sharedScope?: 'parent' | 'officer' | 'child'`
  - `kind === 'shared'` のときに使う
  - `kind === 'private'` では持たない
- `title: string`
- `completed: boolean`
  - `createdAt: string`（ISO8601形式。並び順・履歴管理の基準とする）
  - `memo?: string`（任意。複数行自由入力）
  - `createdByUid?: string | null`
- `assigneeUid: string | null`
  - `null` = 未アサイン
  - `uid` = 担当者（担当は1人のみで開始）
- `dueDate?: string`（任意、日付のみ `YYYY-MM-DD`。時刻は持たない）
- `related?: { type: 'event' | 'session', id: string } | null`（任意）
  - `related` は最大1つ（`event` か `session` のどちらか1つ）
  - `null` / 未設定 = 紐づきなし（Inbox的TODO）
  - Inbox的TODO = 特定のイベント/セッションに紐づかない全体タスクを指す。

### 24.3 TODOページ（メニュー）の表示ルール

- TODOページは `共有TODO` / `個人TODO` のタブ切替方式とする。
- `共有TODO` タブでは、既存の一覧・フィルタ・未アサイン / アサイン済み表示を維持しつつ、ログインユーザーが閲覧可能な共有範囲のTODOだけを表示する。
- `個人TODO` タブでは、本人の個人TODOだけを対象に、必要最小限の一覧と完了管理を行う。
- TODO 一覧ではタイトルを主表示とし、メモは一覧密度を壊さない範囲でプレビュー表示に留めてよい。全文を常時大きく表示しすぎない。
- 利用可能な TODO 種別が 1 種類しかないロールには、`共有TODO / 個人TODO` の切替UIを表示しない。
- `teacher` / `obog` は共有TODOを扱わず、個人TODO一覧のみを表示する。
- 未アサインTODOを優先的に表示するが、アサイン済みも表示する（全体が見えること）
- 推奨並び（提案）
  - セクション1: 未アサイン（`dueDate` 昇順 → `createdAt` 昇順）
  - セクション2: アサイン済み（`dueDate` 昇順 → `createdAt` 昇順）
- フィルタ（最小）
  - 状態: 未完了 / 完了
  - 担当: すべて / 未アサイン / 自分（将来: 任意ユーザー）
  - 紐付け: すべて / セッション / イベント / なし
- TODO追加 / 編集モーダルは、誤タップ防止のため背景クリックでは閉じず、`×` または `キャンセル` で閉じる。
- TODO追加 / 編集モーダルの担当候補は、実Memberデータをメンバー画面デフォルト順に従って表示する。
- TODO追加 / 編集モーダルの紐付け先候補は、`予定` / `イベント` ともに本日以降の候補に絞る。
- TODO追加 / 編集モーダルでは `共有TODO / 個人TODO` を選べるようにする。
- `共有TODO` 選択時のみ、担当・紐付け種別・紐付け先の入力を表示する。
- `個人TODO` 選択時は、タイトル・期限を中心にしたシンプル構成とする。
- TODO追加 / 編集モーダルには、タイトルに加えて任意のメモ欄を持たせる。メモ欄は textarea 等の複数行入力を基本とする。
- 担当プルダウンの固定選択肢 `自分` は置かず、実Member一覧の中で自分の名前にだけ `（自分）` を付ける。
- `共有TODO` の作成可能範囲は以下とする。
  - `child`: `個人TODO` と `child` 共有TODOを作成できる
  - 一般 `parent`: `個人TODO` と `parent` 共有TODOを作成できる
  - `officer`: `個人TODO` と `parent` / `officer` 共有TODOを作成できる
  - `admin`: `個人TODO` と `parent` / `officer` / `child` の全共有TODOを作成できる
  - `teacher` / `obog`: `個人TODO` のみ作成できる
- `共有TODO` を選んだとき、共有範囲選択UIは複数候補から選べる利用者にだけ表示する。
  - `child`: 共有範囲は `child` に自動決定し、共有範囲UIは表示しない
  - 一般 `parent`: 共有範囲は `parent` に自動決定し、共有範囲UIは表示しない
  - `officer`: `parent` / `officer` を選べる
  - `admin`: `parent` / `officer` / `child` を選べる
  - `teacher` / `obog`: 共有TODOは作成しない
- `共有TODO` の閲覧可能範囲は以下とする。
  - `child`: `child` 共有TODO
  - 一般 `parent`: `parent` 共有TODO
  - `officer`: `parent` / `officer` 共有TODO
  - `admin`: `parent` / `officer` / `child` の共有TODOすべて
  - `teacher` / `obog`: 共有TODOなし
- `共有TODO` の担当候補は、そのTODOを閲覧できる範囲内から選ぶ。
  - `child` 共有TODO: `child` の中から選ぶ
  - `parent` 共有TODO: `parent` / `officer` / `admin` の中から選ぶ
  - `officer` 共有TODO: `officer` / `admin` の中から選ぶ
  - `個人TODO`: 担当なし

### 24.4 上部TODOアイコン（受信箱）の表示ルール

- `共有TODO` は `assigneeUid == 自分` かつ `completed == false` を表示する（期限が近い順）
- `個人TODO` は `createdByUid == 自分` かつ `completed == false` を表示する
- 受信箱では `個人TODO` と `共有TODO（自分担当）` を混在させず、セクションを分けて表示する
- `共有TODO` は、自分が閲覧可能な共有範囲のものだけを対象にする。
  - `child`: `child` 共有TODOのみ
  - 一般 `parent`: `parent` 共有TODOのみ
  - `officer`: `parent` / `officer` 共有TODO
  - `admin`: 全共有TODO
  - `teacher` / `obog`: 共有TODOなし
- `dueDate` があるものを昇順で表示する。
- `dueDate` が未設定のTODOは、期限ありTODOの後に表示する。
- 同一条件の場合は `createdAt` 昇順とする。
- TODOページでのアサイン変更結果は受信箱に反映されること

### 24.5 アサイン操作（巻き取り）

- 未アサイン: 引き取る → `assigneeUid = 自分`
- アサイン済み: 引き継ぐ → `assigneeUid = 自分`
- 文言は「奪う」ではなく「引き継ぐ」で統一する
- DEMO注記: クイック操作は「自分へ引き取る/引き継ぐ」を提供し、他者への割り当てはTODO編集UIで行う。

### 24.6 紐づきサマリ表示（TODO行）

- `related` がある場合、TODO行に紐付け先のサマリを表示し、クリックで紐付け先へ遷移できる想定
- 表示例（アイコンは要件。具体の絵文字/アイコンはUI実装時に調整可）
  - セッション紐付け: `📅 {日付} {セッション名}`
  - イベント紐付け: `🎪 {イベント名}`
  - 紐付けなし: `—`
- 予定 / イベント詳細などの共有画面に出す関連TODOも、ログインユーザーが閲覧可能な共有範囲のTODOだけを表示する。
  - `child`: `child` 共有TODOのみ
  - 一般 `parent`: `parent` 共有TODOのみ
  - `officer`: `parent` / `officer` 共有TODO
  - `admin`: 全共有TODO
  - `teacher` / `obog`: 共有TODOなし
- 他人の `個人TODO` は、予定 / イベント詳細などの共有画面には表示しない。

## 25. 購入依頼・立替仕様（SSOT）

### 25.0 横断UIルール

- 登録・作成・更新系フォームの送信ボタンは、必須未入力だけを理由に disabled にしない。
- 送信ボタンは押せる状態にし、送信時にバリデーションを行う。
- 必須入力エラーは、該当フィールド単位で表示する。
- 画面上部の総合エラーより、入力欄の近くに出すエラーを優先する。
- 送信中のみ、二重送信防止のため disabled または loading にしてよい。
- 購入依頼・立替の購入日は日付までを扱う。
  - UI入力は日付のみとする。
  - 一覧表示・詳細表示も日付のみを基本とする。
- 購入依頼・立替は、状態に関係なくカードクリックで詳細モーダルを開けるようにする。
  - 詳細モーダルから、権限と状態に応じた編集・削除・状態変更へ進める。

### 25.1 購入依頼（Purchase Requests）

#### 目的

備品や消耗品の不足・欠品に気づいたときに登録し、購入担当者が見落とさないようにするための記録機能。

承認フローは持たない。

#### データ項目（MVP）

- `title`（必須）
- `category`（任意）
- `memo`（任意）
- `quantity`（任意）
- `estimatedAmount`（任意）
- `createdAt`（任意。DEMOでは並び順制御に利用可能）
- `createdBy`（必須）
- `purchaseAssignees`（任意・複数可）
- `status`
- `boughtBy`（`BOUGHT` 時のみ）
- `boughtAt`（`BOUGHT` 時のみ）
- `purchaseResult?`（`BOUGHT` 時の購入完了入力を保持）
  - `itemName`
  - `quantity?`
  - `amount?`
  - `purchasedAt`
  - `receiptFilesMeta?`（DEMOでは画像アップロードしたファイル情報を保持）

#### 状態

- `OPEN`（未購入）
- `BOUGHT`（購入済み）

#### 操作ルール

- 誰でも作成可能
- `OPEN` の間は、作成者本人または管理者が編集可能
- 誰でも `BOUGHT` に変更可能
- 削除は `OPEN` の間は作成者本人または管理者が可能とする
- `BOUGHT` 後は一般作成者の編集・削除は不可とし、管理者のみ編集・削除できる
- 承認・却下ステータスは持たない
- 一覧カードの基本表示は「タイトル / 依頼者 / 依頼日時 / 状態」とする（カテゴリ・担当・数量・見込金額は一覧では基本表示しない）
- `BOUGHT` への遷移は「購入完了モーダル」を経由する
- 購入完了モーダルで以下を記録できる
  - 買ったもの（初期値は依頼タイトル、編集可）
  - 数量（必須）
  - 購入金額（必須）
  - 購入日（初期値は当日）
  - レシート（任意・複数。DEMOでは画像アップロードしてプレビュー表示）
- レシート画像はDEMOでは保存しない（Storage等の永続化は今後実装）
- 購入完了時の追加記録（片方向）
- 一般保護者（`member`）は「立替に記録する」チェック（デフォルトON）
- 管理者/会計担当（`admin`）は「会計に支出を記録する」チェック（デフォルトON）を使って、その場で会計モジュールへ実起票する
- 追加後の追跡・逆同期は行わない
- 管理者/会計担当が購入完了時に会計連携導線を使う場合は、`出金元口座 / 科目 / 摘要` を入力して、その場で会計モジュールへ支出 1 件を実起票する
- 購入依頼から直接会計起票する場合も、会計本体と同じ固定科目体系を使う
- 購入依頼から直接会計起票する場合の科目選択は必須とする
- 実運用上は `消耗品費` または `雑費` になることが多いが、固定せず選択可能とする
- 摘要は必須とし、購入名や依頼内容をもとに初期値を入れてよい
- 同一の購入完了処理で会計支出を重複生成しないよう管理する
- 購入済み詳細モーダルでは、少なくとも `タイトル / メモ / 依頼者 / 購入者 / 数量 / 購入金額 / 購入日 / 添付画像 / 会計連携状態` を確認できるようにする
- 購入依頼の作成時、カテゴリ入力は現時点では採用しない（入力負荷・迷いを増やすため）
- 代わりに、タイトル入力の過去履歴サジェストで再入力を補助する
- 件数が増えて検索性が課題になった場合に、カテゴリ/タグ導入を検討する（将来拡張）

### 25.2 立替（Reimbursements）

#### 目的

会計担当者以外が立替購入した場合の精算管理。精算忘れ防止を目的とする。
立替は単独作成を主とし、購入依頼（Purchase Requests）とは独立モジュールとして扱う。

承認フローは持たない。

#### データ項目（MVP）

- `title`（必須）
- `amount`（必須）
- `purchasedAt`（必須）
- `buyer`（必須）
- `memo`（任意）
- `receipt`（任意）
- `relatedPurchaseRequestId`（任意）
- `paidByTreasurerAt`（任意）
- `receivedByBuyerAt`（任意）

#### 状態

- UI表示は以下の3状態バッジのみを使う
  - 未精算: `paidByTreasurerAt == null`
  - 支払済: `paidByTreasurerAt != null && receivedByBuyerAt == null`
  - 完了: `paidByTreasurerAt != null && receivedByBuyerAt != null`
- `受領待ち` のような追加バッジ文言は使わず、未完了の支払後は `支払済` で統一する

#### 完了ルール

- `PAID_BY_TREASURER` と `RECEIVED_BY_BUYER` が揃ったら `DONE` とする
- `status` は保存値ではなく、`paidByTreasurerAt` / `receivedByBuyerAt` から算出する（UI表示用）
- `DONE` は2つのタイムスタンプが揃った時点で自動判定する

#### 表示・操作ルール（DEMO）

- 表示範囲（ロール別）
  - `member`: 自分の立替のみ表示（`buyer === currentUserId`）
  - `admin`（会計担当）: 全員分を表示（自分の分を含む）
- 立替は単独作成を基本導線とする（購入依頼からの派生作成に依存しない）
  - 作成項目（最小）: `title` / `amount` / `purchasedAt` / `memo?`
  - レシートは任意・複数。購入依頼の「購入完了」モーダルと同じ画像アップロードUI（複数選択・プレビュー・個別削除）を使う
  - DEMOでは画像は永続化せず、選択ファイル情報を表示用メタデータとして保持する
  - `buyer` は作成時の `currentUserId` を自動設定する
- レシートOCRによる入力補助（初版）
  - OCR対象は添付画像の1枚目のみとし、1枚目をレシート画像として扱う想定とする
  - OCRは添付画像選択時に自動開始する（画像未選択時は実行しない）
  - OCR実行中は進行モーダルを表示し、ユーザーは `中止` して手入力へ切り替えできる
  - ライブラリ都合で完全停止できない場合でも、中止後は結果をフォームへ反映しない
  - 1枚目がレシートとして読み取れない場合は失敗扱いとし、無反応にせず `読み取れませんでした` 相当の軽い通知を表示する
  - 成功時の自動反映対象は `amount` のみとする（`title` は自動入力しない）
  - `amount` 候補は実際の支払額を優先して抽出する
  - OCR結果はあくまで補助値とし、最終確定はユーザー操作で行う
  - 初版OCRの対応対象は立替モジュールのみとする
- タブは上部で `未完了 / 完了` の2つ
  - `未完了` = 未精算 + 支払済
  - `完了` = 完了（領収済）
- カードの基本表示順は `購入者 / 購入日 / 金額`
  - 品目タイトル・メモ・領収書・関連購入依頼IDは必要に応じて追加表示する
- お弁当由来の立替は、システム文字列ではなく自然な文言で表示し、カード上ではお弁当アイコン等で由来が分かるようにする
- 操作ボタンと遷移
  - 未精算:
    - `admin` のみ `支払済にする` を表示（`paidByTreasurerAt = now`）
    - `支払済にする` は専用モーダルで確認する
      - `会計に支出を記録する` チェックを表示し、デフォルトはON
      - チェックON時のみ、その場で会計へ支出記録を実起票する（片方向・逆同期なし）
      - チェックOFF時は支払済更新のみ実行する
      - 会計起票時は、少なくとも `出金元口座 / 科目 / 摘要` を指定できるようにする
      - お弁当由来の立替では、会計科目の初期選択を `講師・その他` にしてよい（変更は可能）
    - 立替作成モーダルには `会計に支出を記録する` チェックを表示しない（会計記録の選択は支払済モーダルに集約）
    - 作成者は、未精算かつ未領収の間のみ編集・削除可とする
    - 管理者は状態に関係なく編集・削除できる
  - 支払済:
    - 購入者（`buyer`）のみに `領収済にする` を表示（`receivedByBuyerAt = now`）
    - 領収済確認モーダルでは、少なくとも `タイトル / 金額 / 購入者 / 購入日` を表示する
  - 完了:
    - 操作ボタンなし（閲覧のみ）
- `支払済にする` / `領収済にする` / `削除` は共通の確認ダイアログUIで実行する（`window.confirm` は使わない）
- 立替から直接会計起票する場合も、会計本体と同じ固定科目体系を使う
- 立替の科目選択は必須とする
- 実運用上は `消耗品費` または `雑費` になることが多いが、固定せず選択可能とする
- 摘要は必須とし、立替タイトルをもとに初期値を入れてよい
- 領収済は領収確認や書類確認の完了を表す状態であり、会計起票とは切り離す
- 同一立替に対して会計支出を重複生成しないよう管理する

### 25.3 会計連携ポリシー

#### 基本思想

- 会計との連携は片方向とする
- 周辺モジュールから会計へ実起票し、会計側からの逆同期は行わない
- 会計側で修正・削除されても購入依頼・立替側は影響を受けない
- 会計連携は、各モジュールで初回確定時のみ行う
- 編集時には会計連携を行わない
- 編集画面では、会計関連項目を通常の操作対象にしない
- 編集によって会計内容との不整合が生じうる場合は、必要に応じて会計モジュール側で手動対応する
- これにより、編集時の二重計上や意図しない再連携を防止する
- 周辺モジュールから会計へ直接支出レコードを追加する場合、少なくとも `出金元口座 / 日付 / 金額 / 科目 / 摘要` を必須とする
- 必要に応じて、会計レコードに元モジュールの出所情報を保持してよい
- 同一の業務レコードから同じ会計支出を複数回生成しない前提で管理する
- 会計連携導線は「準備情報の保存」ではなく「実起票」とする
- 起票成功時のみ `accountingRequested` / `accountingLinked` / `accountingEntryId` / `accountingSourceType` / `accountingSourceId` / `accountingAccountId` / `accountingCategoryId` / `accountingMemo` などのリンク情報を保存する
- 起票失敗時は `accountingLinked=true` にしない

#### 直接会計起票時の共通ルール

- 出金元口座の指定を必須とする
- 口座選択肢は当該会計期で利用可能な口座一覧とする
- 口座一覧の並び順は `sortOrder` 昇順、同値または未設定時は `name` 昇順、最後に `id` で安定化する
- 初期値はログインユーザーの役割に応じて自動選択してよい
  - 会計担当者: `現金（会計手元金）`
  - 管理者: `現金（会長手元金）`
- 初期値は固定ではなく、ユーザーが変更可能とする
- 初期値に対応する口座が存在しない場合は未選択で開始し、ユーザーに選択させる
- 周辺モジュールから直接会計起票する場合も、会計本体と同じ固定科目体系を使う
- 摘要は必須とする
- 摘要初期値は元モジュールの内容から自動生成してよいが、ユーザーが編集可能とする

#### 連携ルール

- 購入依頼で直接会計へ記録する場合は、購入完了時に会計へ支出 1 件を実起票する
- 立替で会計へ記録する場合は、`支払済` にする時点で会計へ支出 1 件を実起票する
- お弁当で直接会計へ記録する場合は、登録時に会計へ支出 1 件を実起票する

### 25.4 Firestore 正式化

- `minamigaoka` の購入依頼・立替・お弁当は、DEMO 時代のローカルモック配列を正データとして扱わず、Firestore を正として表示・追加・更新・必要な削除を行う。
- Firestore コレクションは少なくとも `purchaseRequests` / `reimbursements` / `lunchRecords` を使う。
- 読み込み中は無反応に見せず、空状態とエラー状態を画面内で自然に表示する。
- 既存 UI のカード・一覧・モーダル導線は極力維持し、保存先だけを Firestore に置き換える。
- 会計連携の将来実装に備えて、購入依頼・立替・お弁当の各レコードは少なくとも以下の情報を保持できる構造にする。
  - `accountingRequested` または同等の、会計起票を実行した状態
  - `accountingLinked`
  - `accountingEntryId`
  - `accountingSourceType`
  - `accountingSourceId`
  - `accountingAccountId`
  - `accountingCategoryId`
  - `accountingMemo`
- 購入依頼やお弁当から立替を派生させる場合は、重複生成を避けるために `reimbursementLinked` / `reimbursementId` または同等の参照を保持してよい。
- 出金元口座の候補順は会計モジュールの口座並び順ルールに従う。
- 購入依頼と立替の科目は会計本体と同じ固定科目体系を前提とし、お弁当の直接会計起票時は `講師・その他` 固定を前提とする。
- 会計は、団体の財布・口座が実際に動いた事実のみを記録する。
- 購入依頼・立替・お弁当から会計への連携は片方向とし、逆同期は行わない。
- 重複起票防止のため、`accountingLinked` / `accountingRequested` / `accountingEntryId` などの状態を保持して管理する。

- 一覧は画像中心のグリッド（インスタ風）とし、画像なしはプレースホルダー表示を行う。

## データモデル基本構造（Schedule × Event × TODO）

Windomsにおける中核モデルは、以下の3要素で構成される。

- **Schedule（スケジュール / セッション）**
- **Event（イベント / 上位概念）**
- **TODO（タスク）**

この3つは役割が明確に分かれており、責務が重ならない設計とする。

---

### 1. Schedule（セッション単位）

Scheduleは「日時を持つ最小の活動単位」である。

例：
- 2/18 練習（18:00-21:00）
- 合宿1日目 午前練習
- 本番当日リハーサル

#### 特徴

- 必ず日時を持つ
- Today表示の主役
- Eventに0または1で紐づく
- TODOを複数持てる

Scheduleは「時間軸の単位」であり、概念のまとまりではない。

---

### 2. Event（上位概念）

Eventは「目的を持つまとまり」である。

例：
- 定期演奏会
- 夏合宿
- アンサンブルコンテスト
- 保護者説明会

#### 特徴

- 日付レンジを持てる
- 複数のScheduleを内包できる（1対多）
- TODOを直接持てる

Eventは「物語・プロジェクト単位」であり、時間の最小単位ではない。

---

### 3. TODO（タスク）

TODOは「やるべきこと」の単位である。

例：
- 楽譜印刷
- 見守り入力
- 当番申請
- 事前練習参加確認

#### 特徴

- 単体でも存在可能
- Scheduleに紐付け可能
- Eventに紐付け可能
- 両方に同時紐付けはしない（設計単純化のため）

TODOは責務単位であり、時間やプロジェクトの概念ではない。

---

## 関係性ルール

| 関係 | ルール |
|------|--------|
| Event → Schedule | 1対多 |
| Schedule → Event | 0または1 |
| Schedule → TODO | 1対多 |
| Event → TODO | 1対多 |
| TODO → Schedule | 0または1 |
| TODO → Event | 0または1 |

※ TODOはScheduleとEventの両方に同時紐付けしない。

---

## 合宿パターン例

例：夏合宿

Event：
- 夏合宿2026

Schedule：
- 1日目 午前練習
- 1日目 午後練習
- 2日目 午前練習

TODO：
- 持ち物確認（Event紐付け）
- 1日目見守り（Schedule紐付け）
- 係分担決定（Event紐付け）

---

## 削除ポリシー（暫定）

- Event削除時：
  - 紐付くScheduleはEventなし状態に戻す（削除しない）
  - 紐付くTODOも同様に孤立させる

- Schedule削除時：
  - 紐付くTODOは孤立させる

物理削除ではなく、構造の崩壊を避ける。

---

## 設計思想まとめ

- Scheduleは時間軸
- Eventは物語軸
- TODOは責務軸

この三軸が交差することで、Windomsはクラブ運営の全体像を表現する。

## 29. Instruments Module (DEMO v0)

- `一般公開 = OFF` の場合は、そのモジュールを管理者以外に非表示とする。
- `一般公開 = OFF` でも、管理者にはそのモジュールを表示する。
- `一般公開` を OFF にしても、個別の権限設定値自体は保持する。
- `一般公開` を再度 ON に戻した場合は、保持していた個別権限設定をそのまま再利用する。
- モジュール管理画面は、カード縦積みではなく一覧表形式で表示する。

- `一般公開` 列は、基本的に `モジュール` 列の右隣へ配置する。
- モジュール名にはアイコンを表示する。
- カテゴリ（活動 / 会計 / 資産 / 設定）は、可能な範囲で区切り行として残す。
- モジュール管理画面の表では、列ヘッダを固定し、縦スクロール中でも列見出しを見失わないようにする。
- モジュール管理画面では、表エリア内に専用の縦スクロール領域を設ける。
- 通常の画面幅とは別に、PC 利用を主対象とする管理画面向けの広幅レイアウトを用意する。
- この広幅レイアウトは、表・マトリクス・一覧比較型 UI を持つ画面に適用できる共通レイアウトとする。
- 初回適用対象は `モジュール管理` 画面とし、将来的に `シフト作成` などの PC 中心画面にも再利用できる形とする。
- モジュール管理画面は、ページ全体の縦スクロールを維持しつつ、必要に応じて横スクロールを許容する。
- 列ヘッダは、その表内スクロール領域に対して固定する。
- 表内スクロールで先頭行から末尾行まで確実に到達できることを前提とする。
- 画面幅が不足する場合は横スクロールを許容し、比較しやすさを優先する。
- スマホでは無理に詰め込まず、必要に応じて横スクロールを許容する。
- 実際のメニュー表示は、その設定結果に従って出し分ける。
- 実際のメニュー表示判定では、まず `一般公開` を確認し、`OFF` の場合は管理者以外に表示しない。
- 各モジュール説明にある「子ども向け / 保護者向け / 管理者向け」等の記述は、固定制約ではなく運用上の想定・思想として扱う。
- 固定制約としてコードで強制するのではなく、実際のメニュー表示可否はモジュール管理設定を正として決める。
- 旧DEMO由来の決め打ち `role` フィルタには依存しない。
- ただし、`モジュール管理` と `メンバー管理` は管理者の再入場確保のため固定の管理者向けモジュールとして扱う。
- `一般公開 = OFF` のモジュール行は、モジュール管理画面上で軽くグレーアウト表示し、公開されていないことが視覚的に分かるようにする。
- ただしグレーアウトは見た目のみとし、個別の権限設定操作自体は可能なままとする。
- 初期状態は既存利用者の導線を壊さない安全な既定値とし、既存メニューが一斉に消える状態を作らない。
- 既存データに `一般公開` の保存値が無い場合は、後方互換のため `ON` として扱う。
- 管理者には `モジュール管理` を固定表示し、モジュール管理画面上で変更不可とする。
- 管理者には `メンバー管理` も固定表示し、モジュール管理画面上で変更不可とする。
- 管理者が `メンバー管理` `モジュール管理` に再入場できなくなる設定は回避する。
- モジュール管理画面の表は、`一般公開` 列追加後も既存の固定ヘッダ・横スクロール・縦スクロール前提の UI を維持する。

### 30.9 Phase 2: family / member / memberRelation 基盤

#### 30.9.1 データモデルの基本方針
- `family` は家庭単位のまとまりを表す。
- `member` は個人を表す。
- `memberRelation` は個人同士の関係を表す。
- `family` は生活単位、当番単位、車両所有単位の基盤とする。
- `member` はログイン主体、権限主体、会費対象、出欠や日誌の主体とする。
- `memberRelation` は続柄などの関係性を表す。
- `member` は原則として 1 つの `family` にのみ所属する。
- 血縁や関係性が複雑でも、`family` の複数所属ではなく `memberRelation` で表現する。

#### 30.9.2 ロールと権限
- 正式仕様の権限モデルは単一 `role` ではなく、`memberTypes` / `adminRole` / `staffPermissions` / `memberStatus` を基準にする。
- 既存の単一 `role` は暫定互換のために残り得るが、新規設定・今後の判断基準は本節の新構造を優先する。

#### 30.9.2.1 正式仕様の権限モデル
- `memberTypes: string[]`
  - 利用者の立場を表す
  - 複数保持可
  - 当面の候補値は `parent` / `child` / `teacher` / `obog`
- メンバー種別の UI 表示名は以下に統一する。
  - `parent = 保護者`
  - `child = 部員`
  - `obog = 先輩`
  - `teacher = 先生`
- `adminRole: string`
  - クラブ全体に対する管理・運営権限を表す
  - 単一値
  - 候補値は `none` / `officer` / `admin`
- `staffPermissions: string[]`
  - 特定業務の担当権限を表す
  - 複数保持可
  - 当面の候補値は `accounting` / `shift_management`
- `memberStatus: string`
  - 現在利用対象かどうかを表す
  - 単一値
  - 候補値は `active` / `inactive`
- `memberTypes` は「その人が誰か、どんな立場を持つか」を表す。
- `adminRole` は「クラブ全体に対する管理・運営権限」を表す。
- `staffPermissions` は「会計担当やシフト作成担当など、特定業務の担当」を表す。
- `memberStatus` は「現在利用対象かどうか」を表す。
- `admin` は原則すべて利用可能な最上位管理者とする。
- `officer` は運営系の追加権限とする。
- `accounting` は会計モジュール利用、購入依頼に対する購入、立替に対する支払いなど、会計担当として通常必要な業務を一通り含む。
- `shift_management` はシフト作成や調整に必要な通常業務を含む。
- 今回は `accounting` / `shift_management` をさらに細分化しない。

#### 30.9.2.2 memberTypes の組み合わせルール
- `memberTypes` は複数可だが、当面は以下の組み合わせ制約を設ける。
- `parent + obog` は可。
- `teacher + obog` は可。
- `child` は当面単独運用寄りとし、他の `memberTypes` と併用不可。
- `child + parent` / `child + teacher` / `child + obog` は不可。
- UI 上で禁止組み合わせは選択できないようにする。
- 保存時にも同じルールでバリデーションする。
- エラーは該当セクション内に表示し、無反応に見える状態を作らない。

#### 30.9.2.3 旧 role からの暫定読み替え
- 旧 `role` 実装が残っている場合は、互換運用のために以下の読み替えを行う。
- `admin` -> `memberTypes=[parent]`, `adminRole=admin`, `staffPermissions=[]`
- `officer` -> `memberTypes=[parent]`, `adminRole=officer`, `staffPermissions=[]`
- `parent` -> `memberTypes=[parent]`, `adminRole=none`, `staffPermissions=[]`
- `child` -> `memberTypes=[child]`, `adminRole=none`, `staffPermissions=[]`
- `teacher` -> `memberTypes=[teacher]`, `adminRole=none`, `staffPermissions=[]`
- この読み替えは暫定互換のためのものであり、今後の正式仕様は新構造を基準にする。
- `role` は人の立場や属性を表す。
- `permission` は画面や操作の可否を表す。
- `role` と `permission` は分離する。
- 初期 `role` 候補は `admin` / `officer` / `parent` / `child` / `teacher` とする。
- `permissions` は `string[]` で持つ。
- 2026-04 の仮運用時点では `admin` のみ管理画面へアクセス可能とする。
- `officer` 以下の細かい権限制御は将来拡張とする。

#### 30.9.3 認証紐付け
- Windoms 側の正は Firestore 上の `members` とする。
- Firebase Auth はログイン基盤とする。
- `members.authUid` で Auth ユーザーとメンバーを紐付ける。
- `members.loginId` を持つ。
- `members.authEmail` には内部用メールアドレスまたは Auth 側の email を保存する。
- 内部用メールアドレスは `[loginId]@minamigaoka.windoms.club` 形式とする。
- Auth ユーザーの作成 UI とパスワード再発行 UI は今回未実装とする。
- 当面は Firebase コンソールから管理者が手動で Auth ユーザー作成・パスワード再発行を行う。
- 認証済み利用者のアプリ内ユーザー解決は Firestore `members` を正とする。
- ただし既存ログイン成功状態を壊さないため、Phase 1 の `LOGIN_USER_MAP` 的な暫定処理は Firestore `members` 未紐付け時の fallback として当面残してよい。

#### 30.9.4 family
- `families` コレクションを導入する。
- `family` は家庭単位のまとまりである。
- 当番や車両は将来 `family` 単位で扱う。
- `member.familyId` は 1 つのみ持つ。
- `family` の複数所属は今回は扱わない。
- `family` は表示用の短い名称として `name` を持つ。
- `family.name` はカレンダーや当番表示でそのまま使うため、原則として `田中` のような苗字のみを登録する。
- `family` は任意で `address` を 1 本の文字列として持てる。
- `family` は任意で複数台の車両情報 `vehicles` を持てる。
- 車両情報の最小項目は以下とする。
  - `maker`
  - `model`
  - `capacity`
  - `notes`
- `capacity` は乗車定員として数値で扱える形を基本とする。
- `address` と `vehicles` は管理者向けメンバー管理画面でのみ扱い、一般画面には表示しない。
- Family 編集では、車両は Family 本体フォームの中に詳細フォームを縦に並べ続けず、一覧 + 追加/編集導線で扱う。
- 車両追加 / 編集は、Family 編集モーダルとは分離した別モーダルで 1 台単位に行う。
- Family 編集モーダルと車両追加 / 編集モーダルは、縦長になっても操作不能にならないよう最大高さと縦スクロールを持てるようにする。
- `family.name` は表示用の短い名称として扱う。
- カレンダーや当番表示では、この `family.name` をそのまま利用する。
- `family.name` は原則として苗字のみを登録する。
- 例: `田中`
- `田中家` のような家付き表記や、絵文字・装飾付きの名称は登録しない想定とする。
- 今回は機械的な文字種制限ではなく、運用ルールとして明文化し、管理 UI の注意書きでも案内する。

#### 30.9.5 memberRelation
- `memberRelation` は個人同士の関係を表す。
- 1 人の子どもに対して複数の保護者 relation を持てる。
- 続柄は `relationType` の属性として持つ。
- 続柄は `member` 本体の属性ではなく、関係ごとの属性として持つ。
- 例として、ある人が A 君の `mother` であり、B さんの `aunt` であることを表現できる。
- 続柄は子ども視点の意味で扱う。
- 向きは `childMemberId -> guardianMemberId` に統一する。
- 逆方向 relation は保存しない。
- `relationType` 初期候補は `father` / `mother` / `grandfather` / `grandmother` / `uncle` / `aunt` / `guardian_other` とする。
- 管理画面では続柄を日本語表示できるようにしてよい。
- 当面は同一 `family` 内の relation のみ扱う。

#### 30.9.6 未紐付けと不整合
- Windoms に存在するが Auth 未紐付けの `member` を可視化する。
- Auth に存在するが `member` 未紐付けのユーザーを可視化する。
- `memberRelation` や Auth 紐付けの不整合も可視化できるようにする。
- 不整合の例:
  - `members.authUid` はあるが対象 Auth が存在しない。
  - `memberRelation.childMemberId` または `memberRelation.guardianMemberId` の参照先が存在しない。
  - 同じ `childMemberId + guardianMemberId` の重複 relation がある。
- Phase 2 ではまず一覧で分かることを優先し、複雑な自動修復は行わない。

#### 30.9.7 Phase 2 の位置づけ
- Phase 2 は `family` / `member` / `memberRelation` の基盤導入、管理者画面、Auth 紐付け、不整合可視化を目的とする。
- 車両管理の本実装は将来フェーズとする。
- スケジュール / Today / カレンダー / シフトの Firestore 永続化は後続フェーズとする。
- ただし当番日誌は先行対象とし、日単位データの読込・保存を Firestore で扱う。

#### 30.9.9 loginId / authEmail の制約
- `loginId` は `member` に保持する。
- `loginId` は trim + 小文字化して扱う。
- `loginId` は Windoms のログイン用 `userId` として扱う。
- `loginId` / `userId` に使える文字は半角英小文字 `a-z` / 数字 `0-9` / アンダーバー `_` / ハイフン `-` のみとする。
- `loginId` / `userId` の文字数は 4〜20 文字とする。
- `loginId` / `userId` は英小文字で統一する。
- `loginId` / `userId` は重複不可とする。
- `loginId` / `userId` は内部メールアドレス生成の元値として使う。
- 内部用メールアドレス生成は既存の共通関数を再利用する。
- `authEmail` は内部用メールアドレスまたは Auth 側 email を保持する。
- `loginId` の一意性は Phase 2 では管理画面での保存時バリデーションで担保する。
- Firestore の厳密な一意制約は今回未実装とし、将来必要であれば Functions / transaction で強化する。

#### 30.9.10 管理者画面
- `minamigaoka` 側に管理者専用のメンバー管理画面を追加する。
- 最低限以下を扱えるようにする。
  - `family` 一覧
  - `family` 追加 / 編集
  - `member` 一覧
  - `member` 追加 / 編集
  - `member` の `family` 所属設定
  - Auth 紐付け状況の一覧表示
  - `memberRelation` 一覧
  - relation 追加 / 編集 / 無効化
  - 未紐付け / 不整合の可視化
- アクセス制御は `admin` のみとし、UI 非表示だけでなくルート遷移時もガードする。
- `admin` 判定は Firestore `members.role === "admin"` を正とする。
- 軽微な結果通知はローカルトーストでよい。
- 一覧は分かりやすさ優先とし、過剰な作り込みは行わない。

#### 30.9.10.2 メンバー管理の削除ルール
- `memberRelation` は削除可能とする。
- `member` は削除可能とする。
- `member` 削除時は、その `member` を参照している `memberRelations` も一緒に削除する。
- `family` は所属 `member` が 0 件のときのみ削除可能とする。
- `family` に所属 `member` が 1 件以上ある場合、管理画面の Family 一覧では削除ボタンを非活性で表示し、常時の警告文は出さなくてよい。
- Firebase Authentication のユーザー削除は今回の管理 UI の対象外とする。
- 削除操作はすべて確認モーダル経由とする。
- 削除失敗時は対象カード付近またはモーダル内に理由を表示する。
- 削除処理中は無反応に見えないよう、処理中状態を表示する。

#### 30.9.10.3 「メンバー」と「メンバー管理」の役割分担
- 「メンバー」は一般利用者向けの閲覧モジュールとする。
- 「メンバー管理」は設定配下の管理者専用モジュールとする。
- 両者は同じ member 系データを参照するが、責務と UI は分ける。
- 「メンバー」は見る画面、「メンバー管理」は整える画面として扱う。
- 「メンバー」は DEMO にある一般利用者向けの完成形イメージを基準とする。
- 「メンバー」は family / Auth / relation 不整合チェックなどの管理機能を持たない。
- 「メンバー管理」では `members` / `families` / `memberRelations` / Auth 紐付け / role / permission / 有効無効 など、member 系基礎データを扱う。
- 管理画面で整備した情報は、「メンバー」画面だけでなく他モジュールの人物参照にも利用される。
- 一般向けメニューには「メンバー」を置き、「メンバー管理」は設定配下に置く。
- 一般ユーザーには「メンバー管理」を表示しない。
- 管理者のみ設定内で「メンバー管理」に到達できるようにする。
- 画面名やタイトルもこの責務に合わせて整理する。

#### 30.9.10.4 表示名設定
- クラブ名やサイト表示名は JSX 内でハードコードせず、フロント側の設定値で一元管理する。
- 画面内表示と `document.title` は同じ設定値を参照する。
- 文言変更のたびに JSX を直接探して直さなくてよい構成にする。
- 表示名は env ではなく、`src/config` 配下の公開設定ファイルで管理する。
- 設定ファイルには少なくとも `productName` / `organizationName` / `fullDisplayName` / `documentTitle` 相当を持たせる。
- minamigaoka の今回の表示名は `Windoms 南ヶ丘中学校吹奏楽クラブ` とする。
- `productName` は `Windoms`、`organizationName` は `南ヶ丘中学校吹奏楽クラブ` としてよい。
- `document.title` の基本形式は `{ページ名} | {fullDisplayName}` とし、ページ名を持たないトップ相当のみ `fullDisplayName` 単独表示としてよい。
- 例: `ログイン | Windoms 南ヶ丘中学校吹奏楽クラブ` / `メンバー管理 | Windoms 南ヶ丘中学校吹奏楽クラブ`
- `documentTitle` も原則としてこの表示名を基準に構成する。

#### 30.9.10.1 minamigaoka のメニュー / アカウント操作
- 上部タスクバー右端はハンバーガーメニューとする。
- ログアウトボタンは上部タスクバーに常設しない。
- ログインユーザー情報とログアウト導線はハンバーガーメニュー内に配置する。
- メニュー内では、画面遷移系メニューとアカウント操作系 UI を分けて見せる。
- ログアウトは独自の共通確認モーダルを経由する。
- ログアウト確認文言はシンプルでよい。
- ログアウト後はログイン画面へ戻す。
- ログインユーザー情報は `member.name` を優先表示し、補助情報として `loginId` と `role` を表示する。
- `member` 未紐付けなどで `name` が取れない場合は fallback として `loginId` を表示してよい。
- スマホ / タブレットでも見切れず操作しやすい位置に配置する。

#### 30.9.10.5 メンバー管理画面の権限設定 UI
- メンバー管理画面では、以下を設定できるようにする。
  - 利用者区分
  - 管理権限
  - 担当業務
  - 利用状態
- UI は次の 4 ブロックを基本とする。
  - 利用者区分: `memberTypes`
  - 管理権限: `adminRole`
  - 担当業務: `staffPermissions`
  - 利用状態: `memberStatus`
- 利用者区分はチェックボックスで設定し、禁止組み合わせは UI 上で選べないようにする。
- 管理権限は `none` / `officer` / `admin` の単一選択とする。
- 担当業務は `accounting` / `shift_management` の複数選択とする。
- 利用状態は `active` / `inactive` の単一選択とする。
- 保存時にも UI と同じルールで再検証する。
- `memberTypes` は 1 つ以上必須とする。
- `child` を含む場合に他の `memberTypes` が同時に選ばれていたら保存エラーとする。
- エラーは alert や browser confirm ではなく、該当セクション内に表示する。
- 一覧や詳細では、利用者区分 / 管理権限 / 担当業務 / 無効状態が軽く分かる表示を許可する。

#### 30.9.10.6 メンバー管理画面のタブ構成
- メンバー管理画面は、以下の 4 タブで構成する。
  - `Family`
  - `Member`
  - `Auth`
  - `未紐付け / 不整合`
- 各タブは、そのデータ種別や用途に応じた一覧と操作を持つ。
- 管理画面では「通常操作」と「点検・不整合確認」を分離し、迷いにくい構成を優先する。
- ページ上部にタイトル、その下にタブを配置する。
- 各タブ内では、見出し・簡単な説明・主操作ボタン・一覧本体を基本構成とする。
- 追加ボタンや再取得ボタンは、関係するタブの中に置く。
- 通常利用の一覧と、点検用の一覧を同じタブに混在させない。
- `Family` タブ:
  - family の一覧、追加、編集、削除を扱う。
  - 主操作は `Family追加` とする。
- `Member` タブ:
  - member の一覧、追加、編集、削除、relation の通常操作を扱う。
  - 主操作は `Member追加` とする。
  - relation の追加や編集・削除は `Member` タブ側で扱う。
  - 一般向けメンバー画面と同様に、種別フィルタを持つ。
  - フィルタ表示順は `すべて / 部員 / 保護者 / 先輩 / 先生` の固定順とする。
  - フィルタ判定は内部値 `child / parent / obog / teacher` で行う。
  - `すべて` 表示時の全体表示順は `部員 → 保護者 → 先輩 → 先生` とする。
  - 各区分の中では、その区分に設定された並び順を使う。
  - 並べ替え UI は区分ごとに絞り込んだときのみ表示し、`すべて` 表示時には並び順は反映するが並べ替え UI は表示しない。
  - 並べ替え UI はドラッグ操作を第一候補とするが、初期実装では迷いにくさを優先して上下移動などの単純な UI でもよい。
- `Auth` タブ:
  - auth の確認、紐付け、再取得など、認証情報関連を扱う。
  - 主操作は `Auth再取得` とする。
  - loginId / authEmail / authUid の確認や Auth 紐付けもこのタブで扱う。
- `未紐付け / 不整合` タブ:
  - family 未所属、auth 未紐付け、member/auth 不整合、relation 不整合などの点検対象を扱う。
  - このタブは通常操作よりも確認・修正対象の把握を優先する。
  - relation の不整合確認はこのタブで扱う。

#### 30.9.10.7 Member の追加項目
- `members` には既存の権限系項目に加えて、以下の項目を持てるようにする。
  - 氏名項目
    - `displayName`
      - 表示互換用に保持する表示名
    - `familyName`
      - 姓
    - `givenName`
      - 名
    - `familyNameKana`
      - 姓のフリガナ
    - `givenNameKana`
      - 名のフリガナ
    - 新規作成・編集では `familyName` / `givenName` を別入力する
    - 新規作成・編集では、カナも `familyNameKana` / `givenNameKana` を別入力する
    - `displayName` は少なくとも `familyName + givenName` をベースに自動生成してよい
    - 当面の利用者向け表示は `displayName` ベースで壊れないことを優先する
    - 既存互換のため、当面は legacy `name` を `displayName` と同値で併存してよい
    - 既存互換のため、当面は legacy `nameKana` を `familyNameKana` と同値で併存してよい
    - 既存データに `familyName` / `givenName` が無い場合は、暫定移行として `familyName = 旧 displayName`、`givenName = ""`、`displayName = 旧 displayName` を採用する
    - 既存データに `familyNameKana` / `givenNameKana` が無い場合は、暫定移行として `familyNameKana = 旧 nameKana`、`givenNameKana = ""` を採用する
    - 既存 `displayName` の機械的な姓・名自動分割は行わない
    - 既存 `nameKana` の機械的な姓カナ・名カナ自動分割は行わない
    - 管理者向けメンバー管理画面の Member カードでは、フリガナは `familyNameKana` と `givenNameKana` を連結したフルネーム表示とする。片方のみある場合は、その値だけを表示してよい
    - 管理者向けメンバー管理画面の child 向け relation セクション見出しは `保護者` とし、続柄説明文は表示しない
    - 少数の既存テストデータは、その後にメンバー管理画面から手修正してよい
  - `enrollmentYear`
    - 入学年度
    - 西暦 4 桁の数値
    - 例: `2024`
    - `child` では必須寄りだが、当面は空欄保存可とする
    - `obog` にも設定可
    - `parent` / `teacher` は通常空欄可
    - 未設定時は `学年未設定` など、未設定であることが分かる表示を許可する
  - `instrumentCodes`
    - 担当楽器
    - `string[]`
    - 複数担当可
    - 楽器候補はマスタ定義から選択する
    - 将来、候補追加や細分化ができる構造にする
  - `birthDate`
    - 生年月日
    - `YYYY-MM-DD` 形式の日付文字列
    - 任意項目とする
    - CSV 取り込みには含めず、メンバー管理画面から手入力・編集する
    - センシティブ情報として扱う
  - `phoneNumber`
    - 電話番号
    - 文字列として保持する
    - 1 人 1 つでよい
    - 厳格な正規化は今回は行わない
    - 管理者向けメンバー管理画面でのみ扱い、一般画面には表示しない
- `member` は原則として `family` に所属するが、仮運用では `familyId` 空欄の未所属 member を許容する。
- `familyId` 空欄の member は管理画面の点検対象として把握できるようにする。

#### 30.9.10.9 Member 管理画面での扱い
- Member 設定モーダルでは、既存の権限設定に加えて以下を設定できるようにする。
  - `enrollmentYear`
  - `instrumentCodes`
- `birthDate` は `input type="date"` 相当の手入力 UI で設定・編集できるようにする。
- `phoneNumber` は手入力の文字列項目として設定・編集できるようにする。
- `enrollmentYear` は西暦 4 桁を想定した数値入力とする。
- `instrumentCodes` はマスタからの複数選択 UI とする。
- `instrumentCodes` の UI はチェックボックス一覧とし、各選択肢は `チェックボックス + 楽器名` を 1 セットで横並び表示する。
- 担当楽器の選択カード UI は、`利用者区分` の選択 UI と同系統の見た目・操作感に揃える。
- 楽器名は単語の途中で改行しない。
- PC 幅では 2 列表示を基本とし、スマホ幅では 1 列表示とする。
- 各楽器項目は `label` 全体をクリックして ON/OFF できるようにする。
- 横スクロールは発生させない。
- 楽器一覧部分は必要に応じて内部スクロール可能とし、モーダル全体の視認性を保つ。
- `担当楽器未設定` は必要に応じて一覧末尾の通常選択肢として表示してよい。
- `複数選択可` は補助情報であり、主情報より目立たせない。
- 一覧表示では、情報過多にならない範囲で `入学年度` と `担当楽器` を軽く確認できる表示を許可する。
- `enrollmentYear` 未設定で学年表示相当が必要な場面では、`学年未設定` と分かる表示を許可する。
- 管理画面内の種別表示は `部員 / 保護者 / 先輩 / 先生` に統一する。
- 従来の `子ども` 表記は、管理画面では `部員` に置き換える。
- 生年月日と年齢は、管理者のみメンバー管理画面で確認できるようにする。
- 年齢は保存せず、`birthDate` から都度計算する。
- 一般利用者向け画面には、生年月日と年齢を表示しない。
- 電話番号も管理者向けメンバー管理画面でのみ扱い、一般利用者向け画面には表示しない。
- メンバーの並び順は、都度 `学年順` や `楽器順` を計算するのではなく、メンバー管理で設定できる区分ごとの並び順を使う。
- 並び順は `部員 / 保護者 / 先輩 / 先生` ごとに独立して保持する。
- 新規追加メンバーは、その人が属する各区分の末尾に入る。
- 今回の優先適用先は、少なくとも一般向けメンバー一覧と出欠表示とする。
- Family 作成・編集では、`address` と `vehicles` を管理できるようにする。
- Family 一覧では、住所や車両情報を全文表示しなくてもよく、住所の省略表示や `車両 N台` のような要約表示でよい。
- Family カードの車両欄では台数のみを表示し、台数クリックで車両詳細モーダルを開けるようにする。
- 車両詳細モーダルでは、メーカー / 車種 / 乗車定員 / 備考を確認できる。
- 車両が 0 台の場合は詳細リンクを出さない。
- 管理者向けメンバー管理画面の Family 一覧は、PC / スマホともに 1 行 1 カードの縦積み表示とする。
- Family カードの住所は、管理者向け画面内では Google マップ検索へのリンクとして表示してよい。
- Family 編集モーダルでは、車両は `メーカー / 車種 / 乗車定員` が分かる要約一覧を表示し、各車両ごとに `編集 / 削除` を行えるようにする。
- 車両詳細の入力は、Family 本体編集とは分離した別モーダルで行う。
- Family 一覧には、Member と同様の並び順設定 UI を持たせる。
- Family の表示順は、一時的なソート UI ではなく、保存される管理値で制御する。
- Family を選択肢 / 候補一覧 / 一覧表示として出す UI では、原則としてこの保存済み並び順を反映する。並び順が同値または未設定の場合は、表示名の昇順で安定化してよい。
- `address` / `vehicles` / `phoneNumber` は今回 Member CSV 取り込みには含めない。
- これらの Member / Family カード表示仕様は管理者向けメンバー管理画面のみのものであり、一般画面には適用しない。

#### 30.9.10.12 出欠表示と予定詳細
- Calendar と Today の予定詳細表示は、情報構成を可能な範囲で揃える。
- 少なくとも `日付 / 時刻 / タイトル / 出欠集計 / 関連TODO / 場所` の出し方を揃える。
- 関連TODO セクションは、関連TODO が 1 件以上ある場合のみ表示する。
- 関連TODO が 0 件の場合は、`関連TODO はありません` などの文言を出さず、見出しごと表示しない。
- 出欠表示と出欠集計の対象は `部員のみ` とする。
- 保護者 / 先輩 / 先生は、出欠表示および出欠集計の対象に含めない。
- 出欠データが 1 件も存在しない場合でも、対象となる部員は全員分表示する。
- 出欠未入力は `ー` で表示する。
- 出欠表示の状態は `出席 = ○` / `遅刻・保留等 = △` / `欠席 = ×` / `未入力 = ー` とする。
- 出欠集計には `○ / △ / ×` に加えて、未入力数 `ー` を含める。
- 出欠一覧の表示名はフルネームとする。
- ただし現行データ構造では、既存の表示名文字列をそのままフルネームとして扱ってよい。

#### 30.9.10.13 Today レイアウトと誕生日表示
- Today 画面は、Demo 時点のレイアウト感を基準に維持・復旧する。
- 少なくとも `日付見出し / 前日・翌日ボタン / 日誌導線 / 記録あり・なし表示 / お知らせ欄 / 予定カード` を自然な配置で表示する。
- Calendar と Today では、その日に誕生日のメンバーがいる場合に小さなケーキアイコンを表示してよい。
- 誕生日演出の対象区分は `部員 / 先輩 / 先生` とし、保護者は対象に含めない。
- 誕生日判定は `birthDate` の月日で行い、年はお祝い判定に使わない。
- Calendar と Today のケーキアイコンは、同じ誕生日モーダルを開く共通導線とする。
- 誕生日モーダルでは、生年月日や年齢は表示しない。
- 誕生日モーダルの表示名には区分ごとの敬称を付ける。
  - 部員: `○○さん`
  - 先輩: `○○先輩`
  - 先生: `○○先生`
- 複数人が同じ日に誕生日の場合も、各人の区分に応じた敬称で一覧表示する。

#### 30.9.10.10 Member CSV インポート / テンプレート
- CSV インポートは Windoms 専用の整形済みフォーマットを対象とする。
- Google フォームの生データを直接読む機能は持たない。
- 管理者が Windoms 用に整えた CSV を取り込む前提とする。
- CSV の氏名列は `familyName` / `givenName` を正とし、1 列のフルネームからの自動姓名分割は行わない。
- CSV のフリガナ列は `familyNameKana` / `givenNameKana` を正とし、1 列のフリガナ全文からの自動分割は行わない。
- CSV 取り込み時の `displayName` は、少なくとも `familyName + givenName` をベースに自動生成してよい。
- CSV 取り込みでは `familyName` / `givenName` / `familyNameKana` / `givenNameKana` を必須とし、どれかが欠けている行はエラーとして扱う。
- `birthDate` は今回の Member CSV 取り込み列には含めない。
- CSV はヘッダ行ありとする。
- 取り込み時はヘッダ名を参照しない。
- 1 行目は無条件でヘッダとしてスキップし、2 行目以降を列順固定で解釈する。
- まずは Member 新規登録向けの一括取り込みを対象とする。
- 管理画面からテンプレート CSV をダウンロードできるようにする。
- テンプレート CSV はヘッダ行のみでよい。
- ヘッダは以下とする。
  - `familyDisplayName,familyName,givenName,familyNameKana,givenNameKana,loginId,memberTypes,adminRole,staffPermissions,memberStatus,enrollmentYear,instrumentCodes,notes`
- 列順は以下とする。
  - 1 列目: `familyDisplayName`
  - 2 列目: `familyName`
  - 3 列目: `givenName`
  - 4 列目: `familyNameKana`
  - 5 列目: `givenNameKana`
  - 6 列目: `loginId`
  - 7 列目: `memberTypes`
  - 8 列目: `adminRole`
  - 9 列目: `staffPermissions`
  - 10 列目: `memberStatus`
  - 11 列目: `enrollmentYear`
  - 12 列目: `instrumentCodes`
  - 13 列目: `notes`

#### 30.9.10.11 Member CSV 一括ユーザー登録
- メンバー管理画面に、管理者向けの `CSV一括登録` 機能を持つ。
- 入力 CSV は Google フォーム回答をそのまま読むのではなく、Windoms 用に整形した専用 CSV を対象とする。
- 今回の専用 CSV では `loginId` 列を Windoms の `userId` 列として扱う。
- この CSV 一括登録では、1 行ごとに `member` 作成に必要な値を検証し、Auth ユーザーと Firestore 側 `member` をまとめて作成する。
- `loginId` 列を Windoms の `userId` として扱い、内部メールアドレス `loginId@minamigaoka.windoms.club` を生成して Firebase Authentication の Email/Password ユーザーを作成する。
- 仮パスワードはシステム側でランダム生成する。
- 仮パスワード平文は Firebase や Firestore に恒久保存しない。
- 仮パスワード平文は、作成直後の結果一覧にだけ表示し、管理者が外部手段で本人へ通知する前提とする。
- 結果は成功 / 失敗を行単位で表示し、少なくとも `userId` / `displayName` / `generatedEmail` / `temporaryPassword` / `status` / `errorMessage` を確認できるようにする。
- 既存 `loginId` / 内部 email / `authUid` / 既存 `member` の重複がある場合は、その行だけ失敗として継続する。
- Auth ユーザー作成はクライアントから直接行わず、Cloud Functions から Firebase Admin SDK を用いて実行する。
- 一括登録を実行できるのは `admin` custom claims を持つ管理者のみとする。
- 今回は初回ログイン時の強制パスワード変更までは実装しないが、将来拡張候補とする。
- 値ルールは以下とする。
  - `familyDisplayName`
    - 空欄可
    - 空欄時は未所属 member として扱ってよい
    - 値が入っている場合は、まず既存 `family.name` と完全一致する family を探して紐付ける
    - 一致する family が無い場合は、その `familyDisplayName` を使って family を自動作成する
    - 同一 CSV 内で同じ `familyDisplayName` が複数回出る場合は、family は 1 回だけ作成し、2 件目以降は同じ family を再利用する
  - `familyName`
    - 必須
    - 姓
  - `givenName`
    - 必須
    - 名
  - `familyNameKana`
    - 必須
    - 姓のフリガナ
  - `givenNameKana`
    - 必須
    - 名のフリガナ
  - `loginId`
    - 必須
  - `memberTypes`
    - カンマ区切り
    - 例: `child` / `parent,obog`
  - `adminRole`
    - 空欄時は `none`
  - `staffPermissions`
    - カンマ区切り
    - 空欄時は空配列
  - `memberStatus`
    - 空欄時は `active`
  - `enrollmentYear`
    - 空欄可
    - 入っていれば西暦 4 桁数値
  - `instrumentCodes`
    - カンマ区切り
    - 空欄可
    - 値は楽器マスタの `code` を使う
  - `notes`
    - 空欄可
- CSV 取り込み時も、既存の `memberTypes` 組み合わせ制約を適用する。
- `instrumentCodes` は楽器マスタに存在する `code` のみ許可する。
- `familyDisplayName` に一致する family が未作成でも、それ自体はエラーにせず自動作成対象として扱う。
- `loginId` 重複はエラーとして取り込みを止める。
- `familyName` / `givenName` / `familyNameKana` / `givenNameKana` の不足行もエラーとして扱う。
- エラーは可能な範囲で行番号つきで表示し、無反応に見える状態を作らない。

#### 30.9.11 Auth 一覧取得
- 管理画面から Firebase Auth のユーザー一覧を参照できるようにする。
- フロントエンドから直接 Auth 一覧を取得しない。
- Admin SDK を使う安全なサーバー側処理を経由する。
- Phase 2 では Cloud Functions を追加し、Auth 一覧取得 API を提供する。
- API は認証済みかつ `admin` 権限のある利用者のみ呼び出せるようにする。
- `admin` 判定は Firebase Auth の custom claims を用いて行い、当面の判定キーは `admin: true` とする。
- `admin` 権限がない場合は `permission-denied` を返す。
- 管理画面では各 `member` に対して Auth ユーザーを手動で紐付けできるようにする。
- 紐付け時は `members.authUid` と `members.authEmail` を更新する。
- `member.loginId` がある場合は `[loginId]@minamigaoka.windoms.club` との一致候補を見つけやすくする。
- 完全自動紐付けは行わず、最終判断は管理者の手動とする。
- Auth 一覧は Firebase Console 上の存在有無と区別して、画面上で `読み込み中` / `取得成功で 0 件` / `取得失敗` を分けて表示する。
- 取得失敗時は、開発者が原因切り分けに使えるメッセージを確認できるようにする。
- UI 上でも最低限の失敗表示を出し、`0 件` と `失敗` を握りつぶさない。
- 取得元の Firebase project / Functions 経路を開発者が確認しやすい状態にする。
- Auth 一覧取得は Cloud Functions 側で Firebase Admin SDK を用いて行う。
- Functions 側で内部例外が起きた場合は、Cloud Functions ログで特定しやすいようにログを残す。
- 返却情報として、秘密情報を含まない範囲で `server projectId` / `functions region` / `errorCode` / `errorMessage` を確認しやすい状態にする。
- `permission-denied` の場合は、画面上では「取得失敗」として表示しつつ、admin 権限不足だと分かるメッセージを出す。

#### 30.9.11.1 admin 権限付与
- admin 権限は Firebase Admin SDK 経由で custom claims を設定する。
- 当面は仮運用のため、CLI もしくは管理用スクリプトで特定ユーザーへ `admin: true` を付与できればよい。
- まずはしぶやんの minamigaoka ユーザー 1 名を admin にできればよい。
- 本格的な admin 管理 UI は今回の対象外とする。
- 運用上は `functions/scripts/set-admin-claim.mjs` を用い、例として `cd functions && npm run claims:set-admin -- --project windoms-minamigaoka --email [loginId]@minamigaoka.windoms.club` のように実行してよい。
- 実行には Firebase Admin SDK が対象 project を操作できる認証情報が必要である。

#### 30.9.11.2 claims 反映
- custom claims 更新後は、再ログインまたは ID token の強制更新が必要である。
- UI 上で、必要なら「権限更新後は再ログインしてください」または同等の補助文を表示してよい。
- 可能なら権限再取得ボタンや token refresh を用意し、反映確認しやすくする。
- 開発向けには、現在ログイン中ユーザーの `admin` claim 有無を確認できる表示を持ってよい。

#### 30.9.11.3 Functions デプロイ
- Auth 一覧取得と Auth 紐付け更新は Cloud Functions 経由で行う。
- `functions/` ディレクトリで `npm install` を行う。
- `functions/` ディレクトリで `npm run build` を行う。
- リポジトリルートで `firebase deploy --only functions` を実行する。

#### 30.9.11.4 minamigaoka Storage CORS
- 購入依頼 / 立替 / お弁当 / 会計の画像アップロードは Firebase Storage を使う。

gcloud storage buckets describe gs://windoms-minamigaoka.firebasestorage.app --format="default(cors_config)"

または `npm run storage:cors` を使ってよい。`-Apply` を付けない実行は dry-run とし、現在設定の確認だけを行う。

#### 30.9.11.5 minamigaoka Storage Rules

#### 30.9.12 不整合チェック
- 最低限次を管理画面で可視化する。
  - `authUid` が空の `member`
  - どの `member` にも使われていない Auth ユーザー
  - `childMemberId` または `guardianMemberId` が存在しない relation
  - `status === "active"` なのに参照先が無い relation
- 同一 `childMemberId + guardianMemberId` の重複 relation も一覧で分かるとなおよい。

#### 30.9.12.1 relation の当面方針
- `memberRelation` は「子 member から見た保護者 member の続柄」を表す。
- 向きは常に `childMemberId -> guardianMemberId` とする。
- 逆方向 relation は保存しない。
- relation の一覧表示も `子 -> 保護者` で統一する。
- 親側から子ども一覧を出したい場合は `guardianMemberId` で逆引きして表示する。
- 現在の relation データは作り直す前提とし、旧 UI 互換や複雑な移行ロジックは持たない。
- 家系図の完全再現ではなく、クラブ運営に必要な保護者関係の表現を優先する。

#### 30.9.12.2 relationType の当面候補
- relationType は固定候補ベースとする。
- 候補は子どもから見た保護者側の属性のみとする。
- 当面の候補は `father` / `mother` / `grandfather` / `grandmother` / `uncle` / `aunt` / `guardian_other` とする。
- `child` / `brother` / `sister` など、逆方向解決や兄弟関係を要する候補は今回の対象外とする。

#### 30.9.12.3 relation 入力 UI
- `memberRelation` は独立した概念ではあるが、操作 UI は child ロールの `member` にぶら下がる形を基本とする。
- relation の作成・編集・削除は child ロールの `member` カードまたは child 編集導線から行う。
- 独立した `relation 追加` 導線は廃止してよい。
- `memberRelations` セクションを残す場合も閲覧補助にとどめ、主導線は `members` セクション配下とする。
- 入力 UI は `保護者` / `続柄` の 2 項目を基本とし、child は文脈で確定させる。
- child ロールの `member` に対してのみ relation を追加できる。
- `parent` / `admin` など child 以外の `member` には relation 追加 UI を表示しない。
- 補助文として「続柄は子どもから見た保護者との関係です」を表示する。
- 同じ `childMemberId + guardianMemberId` の重複 relation は防ぐ。
- 当面は同一 `family` 内の relation を基本とし、`family` をまたぐ relation は許可しない。
- relationType 候補定義・重複判定・表示整形は関数化し、今後の拡張に備える。
- child カードでは relation を `続柄: 名前` のように分かりやすく表示し、その child の保護者関係を一目で把握できるようにする。
- relation が 0 件の child には `未設定` など分かりやすい表示を出す。

#### 30.9.14 カレンダー / DaySheet のセッション個別CRUD
- セッションの基本作成は `シフト作成` モジュールで一括生成する。
- カレンダーは作成済みセッションの確認と個別調整に使う。
- カレンダーからも単発の追加 / 編集 / 削除はできるが、主用途は微調整とする。
- 内部仕様用語・実装用語としては `セッション` を維持する。
- 一般利用者向け UI 表示では、分かりやすさを優先して `セッション` ではなく `予定` を使ってよい。
- DaySheet からのセッション追加 / 編集 / 削除は `admin` に加えて `shift_management` を持つシフト作成担当者も可能とする。
- 権限のない利用者はカレンダー / DaySheet を閲覧のみとし、追加ボタン / 編集ボタン / 削除ボタンを表示しない。
- UI 非表示だけでなく、Cloud Functions 側でも同じ権限条件を必須にする。
- Calendar と Today の予定詳細では、関連TODO は存在時のみ表示し、出欠集計には未入力 `ー` を含める。
- DaySheet では対象日のセッション一覧を確認でき、`admin` またはシフト作成担当者のみ以下の操作を行える。
  - セッション追加
  - セッション編集
  - セッション削除
- 今回の個別 CRUD で扱う種別は `通常練習` / `自主練` / `イベント` の 3 種とする。
- 内部値は既存仕様に合わせて `normal` / `self` / `event` を使ってよい。
- `event` は独立した別オブジェクトではなく `session.type` の一種として扱う。
- `type === "event"` のときは `eventName` を必須とする。
- `type === "normal"` / `type === "self"` のときは `eventName` を不要とする。
- 月カレンダーでは `event` は `開始時刻 + eventName` を優先表示する。
- DaySheet のセッションカード右上バッジは、`event` でも固定で `イベント` と表示する。
- イベント名そのものは本文側で判別できればよく、バッジは種別表示に徹する。
- DaySheet でも `eventName` が自然に分かるように表示する。
- Today でも `eventName` が自然に分かるように表示することを原則とする。
- セッション当番は `family` 単位で管理する。
- カレンダー / DaySheet の個別 CRUD では、当番 / 見守り担当は `family` 一覧から選択する。
- 当番 / 見守り担当の表示名は `family.name` の苗字を使う。
- 将来 `渋谷家` などの表記に拡張する可能性はあるが、今回の表示は苗字のみでよい。
- 当番 / 見守り担当は未割当を許容してよい。
- 表示ラベルは `type === "self"` のとき `見守り`、それ以外の `normal` / `event` のとき `当番` とする。
- 保存構造は可能な限り共通化し、表示ラベルで `当番 / 見守り` を切り替える。
- 場所は固定候補に縛らない自由入力とする。
- 場所入力では、過去のセッションで使用した `location` を候補表示して入力補助できるようにしてよい。
- 場所候補は軽量な補助 UI とし、自由入力を阻害しないことを優先する。
- カレンダー / DaySheet の単発追加・個別調整に使う `startTime` / `endTime` の入力 UI は、追加・編集ともに 30 分単位の select ベースとする。
- 時刻選択肢は `00:00` から `23:30` までの 30 分刻みを基本とする。
- 終了時刻は開始時刻より後のみを許可し、開始時刻以下の終了時刻は UI 上で選べない / 保持しないようにする。
- 既存データに 30 分単位以外の時刻がある場合は、表示と編集を壊さない範囲で既存値も扱ってよい。
- セッション新規追加時、その日に既存セッションが 1 件以上ある場合は、その日の最後のセッションの終了時刻を開始時刻初期値に使う。
- セッション新規追加時の終了時刻初期値は、開始時刻 + 3 時間とする。
- 同日に既存セッションが無い場合の新規追加初期値は `09:00-12:00` とする。
- 今回の個別CRUDで最低限扱う項目は以下とする。
  - `date`
  - `startTime`
  - `endTime`
  - `type`
  - `eventName`（`type === "event"` のとき必須）
  - `location`
  - `assigneeFamilyId`
  - `assigneeNameSnapshot`（当番 / 見守り担当の表示名スナップショット）
  - `note`
- セッション保存先は既存構造 `scheduleDays/{date}/sessions/{sessionId}` を維持する。
- 今回の CRUD で更新対象とする主なフィールドは `order` / `startTime` / `endTime` / `type` / `eventName` / `dutyRequirement` / `requiresShift` / `location` / `assigneeFamilyId` / `assigneeNameSnapshot` / `note` とする。
- カレンダー表示は永続データを優先し、通常表示で DEMO 専用の仮セッションを混在させない。
- 実データが無い場合は、通常表示では素直に空状態として見せる。
## 25. 会計帳票補足

- 収支計算書は、固定科目体系のカテゴリと項目の 2 階層構造をそのまま使う。
- 収入 / 支出とも、カテゴリ行の下に個別科目行を表示する。
- 画面上の表示順は、固定科目定義のカテゴリ順・項目順に従う。
- 収支計算書では、少なくとも `前年度繰越金 / 当期収入合計 / 収入合計 / 当期支出合計 / 次年度繰越金 / 支出合計` の流れが分かるように表示する。
- 収支計算書上部サマリは、収入系と支出系のまとまりが直感的に分かる配置にする。
- 収支計算書の画面構造は、PDF より先に画面で内容確認しやすいことを優先する。
- 収支計算書には帳票出力用の摘要欄を持てる。
- 帳票用摘要は画面上で自由入力・保存できる。
- 帳票用摘要は取引明細の摘要とは別管理とする。
- 帳票用摘要は個別科目行に対応づけて持つ。
- 帳票用摘要の入力 UI は、確認性を保ちつつ過密にならないようにする。
- 帳票用摘要は、少なくとも `periodId / type(income|expense) / categoryId / subjectId / note` を持てる形で保存してよい。
- 保存した帳票用摘要は、将来の PDF 出力でも参照できるようにする。
## 26. UIダイアログ方針

- ユーザー向け UI では、`window.confirm` / `window.alert` / `window.prompt` などのブラウザ標準ダイアログは使わない。
- 確認が必要な操作は、既存 UI トーンに揃えた共通確認ダイアログまたは画面内の既存モーダルで扱う。
- 削除などの破壊的操作は、対象が分かる文言で確認してから実行する。
- モーダルは用途に応じて、モーダル外クリックで閉じるものと閉じないものを分ける。
- 登録 / 編集 / 作成 / 状態変更など、入力や意思決定を伴うモーダルは、モーダル外クリックで閉じない。
- 情報表示 / 詳細閲覧 / 軽いメッセージ表示など、入力を伴わないモーダルは、モーダル外クリックで閉じてよい。
- 削除確認や重要な状態変更確認など、意思決定の最終段階にある確認モーダルは、モーダル外クリックで閉じない。
- `×` / `キャンセル` / `閉じる` などの明示的な閉じる操作は維持し、システム全体で backdrop click の可否をばらつかせない。
## 27. 会費管理 v1
### 27.1 目的
- 会費管理モジュールは、部員ごとの月謝袋配布状況と月謝回収状況を管理するためのモジュールとする。
- 会費管理は請求や領収の現場管理を担い、実際の収入反映は会計モジュールへ連携する。
- 会費管理 v1 では、複雑な請求管理やオンライン決済は行わず、現在の月謝袋運用をそのままデジタル化することを目的とする。

### 27.2 基本方針
- 管理単位は家族単位ではなく部員単位とする。
- 1 レコードは、1 部員の 1 つの徴収期間を表す。
- 会費レコードは `membershipFeeRecords` コレクションで管理する。
- 月額会費の初期値は 4000 円とする。
- 会計年度は 9 月始まりとする。
- 通常の徴収区切りは `9 月〜12 月`、`1 月〜4 月`、`5 月〜8 月` の 3 期とする。
- ただし、途中入部や休部に対応するため、徴収対象月は月単位で柔軟に設定可能とする。
- 未請求月はレコードを持たず、請求中または領収済みの月だけをレコードで表現する。
- 会費管理から会計への連携は、領収時の実起票とする。
- 会計連携は片方向のみとし、逆同期は行わない。
- 起票成功時のみ会計連携済み情報を保存する。
- 重複起票防止を行う。

### 27.3 データ
- `membershipFeeRecords` の 1 レコードは、少なくとも以下を持つ。
  - `memberId`
  - `memberNameSnapshot`
  - `fiscalYear`
  - `monthKeys`
  - `title`
  - `monthlyAmount`
  - `amount`
  - `status` (`requested` / `received`)
  - `requestedOn`
  - `receivedOn?`
  - `createdAt`
  - `updatedAt`
  - `createdByUid`
  - `receivedByUid?`
  - `accountingRequested`
  - `accountingLinked`
  - `accountingEntryId`
  - `accountingSourceType`
  - `accountingSourceId`
  - `accountingAccountId`
  - `accountingCategoryId`
  - `accountingMemo`
- `monthKeys` は `YYYY-MM` 形式とする。
- レコードのタイトル例は `会費（渋谷◯◯ 2026/09〜2026/12）` の形式とする。
- 対象月数から会費金額を算出し、金額は `月額会費 × 対象月数` を基本とする。
- 将来的に保護者向け通知で `請求中` を判定できるよう、請求中状態をデータから判定できる構造を前提とする。
- この保護者向け案内は、会費管理専用の仕組みではなく、ステータスバー通知が Firestore ベースで正式化された後の共通通知基盤に乗せる前提とする。

### 27.4 トップ画面
- 会費管理モジュールのトップ画面は、年度単位の一覧テーブルとする。
- 行は部員一覧とし、既存のメンバー表示順に従う。
- 列は左端にメンバー名、その右側に 9 月から翌 8 月までの 12 か月を表示する。
- 上部に前年ボタンと翌年ボタンを配置する。
- 各月セルには、その月の会費状態をアイコンで表示する。
- 月セルで表現する状態は `未請求`、`請求中`、`領収済` の 3 種類とする。
- データがない年度でも、画面構造は共通とする。

### 27.5 メンバー別モーダル
- トップ画面でメンバー名をクリックすると、そのメンバーの年度内会費状態を操作するモーダルを開く。
- モーダル内では、9 月から翌 8 月までを縦に並べて表示する。
- 各月行には、トップ画面と同じ会費状態アイコンを表示する。
- 月ごとの状態が見やすく、チェックボックスの有無で行の高さや並びが不自然に崩れない構成とする。

### 27.6 状態別挙動
- 未請求月にはチェックボックスを表示し、複数選択可能とする。
- 未請求月を選択した状態で `月謝袋を渡す` を押すと、選択した月を請求中に変更する。
- 請求中月にはチェックボックスを表示しない。
- そのメンバーに請求中の月が 1 つでもある場合、そのメンバーは請求中扱いとする。
- 請求中扱いのメンバーに対するモーダルの主操作は `領収済みにする` とする。
- 領収操作では、その時点で請求中の会費レコードを対象として領収済みに変更する。
- 領収済月は表示のみとし、通常操作の対象外とする。
- モーダルで表示する主操作は、その時点で優先されるものを 1 つだけとする。
  - 請求中の月が 1 つでもある場合は `領収済みにする`
  - 請求中の月がなく、未請求の月がある場合は `月謝袋を渡す`
  - それ以外は状態表示のみ

### 27.7 会計連携
- 会費管理から会計への連携は、領収済みにしたタイミングで行う。
- 会計には収入として実起票する。
- 会計側の科目は `会費` を使う。
- 入金先口座は既存の会計口座から選択可能とする。
- 初期運用では、会長手元金または会計手元金への入金を主に想定する。
- 領収時の会計摘要は `会費（渋谷◯◯ 2026/09〜2026/12）` の形式を基本とする。
- 会費管理経由では、1 人 1 レコードを基本として会計へ起票する。
- 領収時の会計日付は、領収操作を行った当日の日付を使う。
- 会費管理でも、会計連携は初回の領収確定時のみ行い、編集時には会計連携を扱わない。

### 27.8 UI 方針
- トップ画面では、年間の状態が一目で分かることを優先する。
- モーダルでは、対象月の確認と状態変更が迷わず行えることを優先する。
- チェックボックスは未請求月にのみ表示し、表示有無によって行全体の視認性が崩れないようにする。
- ボタンはその時点で取るべき主操作が 1 つ分かる配置にする。
- 無反応に見える状態を作らず、操作結果は画面上で即時に分かるようにする。

### 27.9 v1 で扱わないもの
- 家族単位の会費請求
- 自動請求生成の高度な条件分岐
- オンライン決済
- 自動督促通知
- 日割り計算
- 複雑な減免ロジック
- 領収済みデータの高度な取消・再計上フロー
