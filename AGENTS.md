# AGENTS.md

## このプロジェクトの基本方針
- READMEを唯一の仕様とする
- コードが正ではなくREADMEが正
- 仕様が固まるまで実装を急がない
- 仕様変更はREADMEに先に反映する
- 実装内容が仕様に影響する場合は、README更新もセットで行う
- 設計判断は decisions.md に残す
- 既存仕様を壊さない最小差分を優先する
- 過剰設計は避け、今必要な粒度で固める
- 迷わない導線を優先する
- UIは無反応に見える状態を作らない
- エラーは該当箇所に出す
- 必須入力エラーはフィールド単位で表示する
- ドロップダウンやフォーカスで隠れる位置にエラーを出さない
- 軽微なフィードバックはローカルトーストでよい
- 表示の揃えは読みやすさ優先で整形する

## 現在の開発フェーズ方針
- 4月に向けた仮運用入り準備フェーズとして扱う
- DEMO用のダミー表示を減らす
- ローカル保存や画面内だけの仮状態を減らす
- Firestoreを正として実装を寄せる
- 実データ取得に順次置き換える
- 見た目だけ動く実装より、実運用で使える状態を優先する

## Codexへの基本ルール
- いきなり実装せず、まずREADMEとの整合を確認する
- READMEに未記載で仕様に影響する内容は、先にREADME差分を出す
- 既存仕様を壊さない前提で差分を作る
- 実装は最小差分を優先する
- UI修正でも仕様として残すべき内容はREADMEへ明文化する
- 納品は可能なら git diff 形式で示す
- 変更ファイルごとに何を変えたか短く説明する
- まず方針整理、その後 diff を出す
- 文字化けや改行崩れが疑われる場合は、内容変更より先に原因を切り分ける

## 文字コード・改行コードルール
- Windows環境前提で作業する
- ファイル操作やパス確認は PowerShell を前提にする
- 日本語を含むファイルは文字化けを最優先で回避する
- 既存ファイルの文字コードを推測で変更しない
- 保存前に、対象ファイルの既存の文字コードと改行コードを確認する
- 文字コードや改行コードに問題がある場合は、実装変更と混ぜずに先に切り分ける
- Markdown、JSON、YAML、TypeScript、TSX、JavaScript、CSS、HTML、TXT は UTF-8 BOMなし + LF を基本とする
- CSV は UTF-8 BOMあり + CRLF を基本とする
- PowerShellスクリプトは既存ファイルに合わせるが、Windows PowerShell 5.1 互換が必要な場合は BOM の有無に注意する
- 新規作成時は、上記ルールに従って保存する
- 既存ファイルが別ルールなら、無断で全体変換しない
- 日本語UI文言を含むファイルは、保存後に文字化けしていないことを必ず確認する

## 改行コードと保存時の注意
- 不要な全ファイル改行変換をしない
- 文字コード修正のついでに、無関係な改行差分を大量発生させない
- 既存の差分が大きくなる場合は、原因を説明する
- CSV は Excel 互換を意識し、UTF-8 BOMあり + CRLF を優先する
- README やソースコードは UTF-8 BOMなし + LF を優先する

## Firestore・実データ化の方針
- DEMOのダミーデータをそのまま使い続けない
- まず現在のローカルデータ構造を確認する
- 過不足がないデータは、構造を作り替えずに Firestore に移すことを優先する
- Firestore正式化フェーズでは、ローカル配列を正データとして残さない
- データ取得は、可能な範囲で service や repository に寄せる
- UIコンポーネント内に Firestore ベタ書きを増やしすぎない
- 既存データが未設定でも壊れないように null 安全を確保する
- 後方互換が必要な場合は README に明記する

## UI実装の方針
- 既存画面の見た目ルールと揃える
- 同じ役割のUIは、別物に見えないようにする
- ボタンやバッジは、既存画面の文言と表現をできるだけ揃える
- 右上バッジには状態、本文には内容、というように役割を分ける
- 情報過多で賑やかになりすぎないように整理する
- グレーアウトは見た目だけか、実際の disabled かを明確に分ける
- disabled でない要素を disabled っぽく見せない
- 共通CSSの変更時は、他画面への波及を確認する
- fixed header、スクロール領域、列揃えは崩さない

## ルーティング・導線の方針
- 利用者が迷わないことを優先する
- 入口ごとの役割を混ぜない
- メニュー導線、Today導線、カレンダー導線などは、それぞれの文脈に応じた役割分担を持たせる
- セッションがない日に不自然な編集画面へ飛ばさない
- 未作成でも、必要なら押下時に自動作成して自然に遷移する
- 戻り導線は、見ていた文脈の日付や月に戻れるようにする

## 当番・Familyまわりの方針
- 当番の割当単位は Family とする
- 当番表示や関連UIは、可能な限り Family 単位で整合を取る
- ユーザー本人名と Family 名が混在する場合は、どちらを正とするか README に明記する
- 既存データに旧形式がある場合は、表示側の後方互換を考慮する

## 納品形式
- まず README 更新方針を簡潔に整理する
- 次に実装方針を簡潔に整理する
- その後 git diff 形式で提示する
- 変更ファイルごとに何を変えたか短く説明する
- 不明点がある場合は、README上の根拠と実装上の根拠を分けて書く
- README だけで判明するなら、不要な解析はしない

## 禁止・非推奨
- README未更新のまま仕様変更を確定させること
- 文字化けの疑いがあるまま日本語ファイルを上書きすること
- 既存ファイルの文字コードを無断で一括変換すること
- 無関係な大規模整形や改行変換を混ぜること
- 直せる範囲を超えてUI全体を別デザインに作り替えること
- 仮実装に戻してその場しのぎにすること
- 根拠なしに推測だけで仕様断定すること

## Git運用ルール
- 機能単位で小さく commit する
- 文字化け復旧や不具合修正のときほど、意味単位で commit を分ける
- 実装依頼では、作業完了後に git status を確認し、必要なら README 更新も含めて commit する
- push は行わない
- push はユーザー側で行う
- commit メッセージは feat / fix / docs / chore などの意味単位で分かりやすく付ける
- commit メッセージは Codex 側で適切に考えてよい
- ただし、ユーザーが明示的に commit メッセージを指定した場合はそれに従う
- 調査のみ、確認のみ、レビューのみの依頼では commit しない
- 実装を伴う依頼では、納品時に commit 済みであることを基本とする
- 最後に、実行した commit メッセージと git status の結果を簡潔に報告する
